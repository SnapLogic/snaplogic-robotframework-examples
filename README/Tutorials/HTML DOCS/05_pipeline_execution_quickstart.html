Pipeline Execution: 5-Step Quick Start

Get your SnapLogic Robot Framework testing environment up and running in minutes

This guide walks you through the essential 5 steps to set up and execute your first Robot Framework test against SnapLogic pipelines. Follow these steps to quickly get started with automated testing.

1

Install Docker Desktop

Download and install Docker Desktop for your operating system, start it, and verify the installation:

docker --version
For Windows Users: Follow the additional instructions for running Docker commands with WSL. See the Windows WSL VS Code Setup Guide for essential setup instructions.
2

Clone the Repository

Create a working directory and clone the GitHub repository:

git clone https://github.com/SnapLogic/snaplogic-robotframework-examples
3

Open Project

Download VS Code (or any preferred IDE) and open the project folder, or work directly from terminal:

# Change the directory to {{cookiecutter.project_name}} eval $(make change-dir) # Directory is changed to {{cookiecutter.project_name}}

OR

cd {{cookiecutter.project_name}}
4

Configure Environment

Create a new .env file at the project root level and configure your credentials:

# Copy the example environment file cp .env.example .env

Then edit the .env file with your actual SnapLogic credentials, organization details, and project settings.

Important: Make sure to delete all inline comments in the .env file when copying from .env.example.
5

Build and Execute

Build your test environment using make commands. Choose one of the following cases based on your setup:

Case 1: Using Framework-Managed Groundplex (Local)

This option creates and manages a Groundplex for you automatically:

# Build the Docker containers for all services make start-services # Or run only tools service for external accounts make start-tools-service-only # Run tests with project space setup make robot-run-all-tests TAGS="oracle" PROJECT_SPACE_SETUP=True # For later executions (project space already exists) make robot-run-all-tests TAGS="oracle"

Case 2: Using Existing Groundplex

If you want to use a different/existing Groundplex and skip creation:

# Build the Docker containers make start-services # Run tests using existing Groundplex make robot-run-tests-no-gp TAGS="oracle" PROJECT_SPACE_SETUP=True # For later executions make robot-run-tests-no-gp TAGS="oracle"
Note: When using robot-run-tests-no-gp, ensure your .env file has the correct GROUNDPLEX_NAME pointing to your existing Groundplex.
Clean Start: When you get the latest code with changes to requirement files or compose files, run:
make clean-start # or make clean-start-tools
Build Time: This creates a containerized environment with Robot Framework and all testing dependencies. The build process takes about 2-3 minutes.

You're All Set!

Your SnapLogic Robot Framework testing environment is ready to go.

What Happens After Execution

Services Started

  • Groundplex is launched for SnapLogic pipeline execution
  • Oracle Database is started
  • PostgreSQL Database is started
  • MinIO (S3-compatible object store) is started and pre-configured

Tests Are Executed

  • Robot Framework tests run - look for green "PASS" messages
  • Accounts are created in SnapLogic Org
  • ProjectSpace is created (existing ones with same name are deleted)
  • Project is created
  • Pipeline is imported
  • Triggered task is created and executed

Test Reports Generated

  • Location: ../snaplogic-robotframework-examples/test/robot_output
  • report.html - Comprehensive test execution report with pass/fail status
  • log.html - Detailed execution logs with step-by-step information
  • output.xml - XML output for integration with CI/CD tools
