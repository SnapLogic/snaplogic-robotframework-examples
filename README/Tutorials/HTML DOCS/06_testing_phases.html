Testing Phases

Complete testing lifecycle from development through production deployment

This guide walks you through the complete SnapLogic Robot Framework testing lifecycle, from initial pipeline development through production deployment. Each phase builds upon the previous one to ensure comprehensive test coverage and quality assurance.

1

Development Phase

Pipeline Creation in SnapLogic Designer

Developer Activities

  • Open SnapLogic Designer in DEV environment
  • Create pipeline (e.g., Kafka to Snowflake)
  • Configure Snaps with placeholder accounts
  • Add transformation logic
  • Set up error handling
  • Validate pipeline structure

Outputs

  • Pipeline file: kafka_to_snowflake.slp
  • Pipeline validated in Designer
  • Manual test passed in DEV
  • Ready for test automation

2

Test Data & Files Preparation

Create test data and input files for validation

Test Data Creation

  • Create sample Kafka messages (JSON/Avro)
  • Prepare CSV/Excel files for file-based pipelines
  • Generate test database records
  • Create XML/EDI test documents
  • Prepare API request/response samples
  • Set up mock data for edge cases

Input Files & Validation Sets

  • test_data/kafka_messages.json
  • test_data/snowflake_expected.csv
  • test_data/error_scenarios.json
  • test_data/performance_dataset.csv
  • Schema validation files
  • Expected output templates

Environment Configuration

Set up .env file with all test configurations

Create .env File

# SnapLogic Configuration SNAPLOGIC_ORG=your-org-name SNAPLOGIC_USERNAME=test-user SNAPLOGIC_PASSWORD=encrypted-pass GROUNDPLEX_NAME=test-automation-plex # Client's Test Systems KAFKA_TEST_BROKERS=kafka-test.client.com:9092 SNOWFLAKE_TEST_ACCOUNT=client-test SNOWFLAKE_TEST_WAREHOUSE=TEST_WH SNOWFLAKE_TEST_DATABASE=TEST_DATA # Test Configuration PROJECT_PATH=/org/QA/DataIntegration TEST_TIMEOUT=300 LOG_LEVEL=DEBUG

Configuration Details

  • All credentials in one place
  • Environment-specific settings
  • Client's test system endpoints
  • Framework configuration
  • Test execution parameters
  • Loaded by Docker containers

4

Test Creation with Framework APIs

Write Robot Framework tests using snaplogic-common-robot Uses .env config & test data

Test Implementation

*** Settings *** Resource snaplogic_common_robot/apis.resource *** Test Cases *** Setup Test Environment # Launch Groundplex for test execution ${groundplex}= Start Groundplex ... name=${GROUNDPLEX_NAME} ... node_count=1 Wait Until Groundplex Ready ${groundplex} Create Test Accounts # Uses credentials from .env ${kafka_account}= Create Account ... type=kafka ... properties=${KAFKA_CONFIG} ${sf_account}= Create Account ... type=snowflake ... properties=${SNOWFLAKE_CONFIG} Import and Execute Pipeline ${pipeline}= Import Pipeline ... file=pipelines/kafka_to_snowflake.slp ... project=${PROJECT_PATH} ${task}= Create Triggered Task ... pipeline=${pipeline.id} ${result}= Execute Task ${task.id}

Framework APIs Used

  • Start Groundplex - Launches test Groundplex
  • Wait Until Groundplex Ready - Ensures plex is operational
  • Create Account - Manages test accounts
  • Import Pipeline - Deploys to test env
  • Create Triggered Task - Sets up execution
  • Execute Task - Runs pipeline
  • Validate Results - Checks data flow
  • All use config from .env file

5

CI/CD Pipeline Execution

Travis CI automatically runs tests in QA environment

Automated Steps

  • Git push triggers Travis CI
  • Load .env configuration
  • Start Docker containers:
    • Tools container (snaplogic-common-robot)
    • Local Groundplex
    • Robot Framework runner
  • Connect to client's test systems
  • Execute all test suites

Test Execution

  • Groundplex started once for all tests
  • Accounts created via APIs
  • Pipelines imported and configured
  • Tasks executed with test data
  • Results validated automatically
  • Reports generated (HTML/XML)
  • Slack notifications sent with results

Production Deployment

Deploy validated pipelines to production

Quality Gates

  • All tests passed in QA
  • Performance benchmarks met
  • Data validation successful
  • No errors in pipeline execution
  • Code review approved
  • Security scan passed

Production Steps

  • Deploy approved pipelines only
  • Update with production credentials
  • Configure production schedules
  • Run smoke tests (limited)
  • Monitor pipeline execution
  • No test data in production
Key Points
  • Phase 2 (test data preparation) ensures all validation inputs are ready before environment setup
  • Phase 3 (.env configuration) MUST happen before Phase 4 (test creation)
  • Test data files are used by the framework APIs for pipeline validation
  • The .env file contains all credentials and configurations needed by the framework APIs
  • Framework APIs (snaplogic-common-robot) rely on environment variables from .env
  • Docker containers load both the .env file and test data files
  • This ensures consistent configuration and test data across all test environments
