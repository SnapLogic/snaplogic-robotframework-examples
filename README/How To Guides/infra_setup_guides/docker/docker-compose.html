Understanding the Robot Framework Testing Environment Setup

Overview

This document explains how the Docker Compose setup works for the Robot Framework testing environment. The architecture provides a consistent, isolated, and reproducible testing environment for all team members.

What This Documentation Covers

  • Complete Docker Compose architecture for Robot Framework testing
  • Container build process and dependencies
  • How to set up and run the testing environment
  • Common usage patterns and best practices
  • Troubleshooting guide for common issues

Key Components

The Docker Compose setup consists of:

  • Tools Container - Main container with Robot Framework and all test libraries
  • Database Services - Oracle, PostgreSQL, MySQL, SQL Server, and more
  • Groundplex Container - SnapLogic runtime environment
  • Supporting Services - MinIO (S3), Kafka, ActiveMQ, etc.

Prerequisites

Before getting started, ensure you have:

  • Docker Desktop installed and running
  • Access to the project repository
  • A configured .env file with necessary credentials
  • Basic understanding of Docker and Docker Compose

Architecture Components

The Docker Compose architecture is modular and scalable:

Main Orchestration File

docker-compose.yml - Central configuration that includes:

  • Service definitions for all containers
  • Network configuration (snaplogicnet)
  • Volume mappings for code and test data
  • Environment variable management

Service Containers

Multiple specialized containers for different services:

  • Tools Container - Robot Framework test execution environment
  • Groundplex - SnapLogic runtime for pipeline execution
  • Databases - Oracle, PostgreSQL, MySQL, SQL Server, DB2, Teradata
  • Message Queues - Kafka, ActiveMQ
  • Storage - MinIO (S3-compatible)

Detailed Setup Steps

1. Docker Compose Configuration

File: docker-compose.yml

  • Defines all service containers (tools, groundplex, databases)
  • Specifies build context and Dockerfile location for the tools container
  • Sets up networking between containers
  • Manages environment variables from .env file

2. Dockerfile Build

File: robot/Dockerfile

  • Base image setup with Python
  • System dependencies installation
  • Working directory configuration
  • Copies the requirements file
  • Triggers pip install process

3. Requirements Installation

File: requirements.txt

Usage Guide

Starting the Services

Bring up the tools container and other services:

make start-services

Or using docker-compose directly:

docker compose --env-file .env -f docker-compose.yml up -d tools

Verifying Installation

Check installed libraries:

docker exec snaplogic-test-example-tools-container pip list

Verify specific library:

docker exec snaplogic-test-example-tools-container pip show snowflake-connector-python

Running Tests

Run tests using the tools container:

make robot-run-tests TAGS="snowflake_demo"

Common Commands

Command Purpose
make start-services Start all configured services
make snaplogic-build-tools Build/rebuild the tools container
make robot-run-all-tests Run complete test suite
make snaplogic-stop Stop all containers and cleanup
docker ps View running containers
docker logs [container-name] View container logs

Benefits of This Architecture

1. Consistency Everyone uses the exact same environment
2. Isolation No conflicts with local Python installations
3. Reproducibility Easy to recreate the environment anywhere
4. Version Control All dependencies are tracked in requirements.txt
5. Easy Updates Simply rebuild the container for updates
6. No Local Setup Developers only need Docker installed
7. Scalability Easy to add new services or scale existing ones
8. Platform Independence Works on Windows, Mac, and Linux
9. Fast Onboarding New team members productive in minutes

Why Docker Compose?

Docker Compose provides several advantages for test automation:

  • Service Orchestration - Manage multiple containers with a single command
  • Environment Variables - Centralized configuration through .env files
  • Networking - Automatic network creation and service discovery
  • Volume Management - Easy data persistence and sharing between containers
  • Development Workflow - Quick iterations with container rebuilds

Cost Savings

This architecture reduces costs by:

  • Eliminating environment-related debugging time
  • Reducing onboarding time for new developers
  • Preventing "works on my machine" issues
  • Enabling parallel test execution
  • Minimizing infrastructure requirements

Troubleshooting Guide

Common Issues and Solutions

Issue Possible Cause Solution
Container fails to build Missing requirements.txt or Dockerfile Verify all required files exist in the correct locations
Tests fail to connect to database Database container not running Check docker ps and ensure database containers are up
Permission denied errors Volume mount permissions Check file permissions and Docker Desktop file sharing settings
Out of memory errors Docker memory limit too low Increase Docker Desktop memory allocation in settings
Port already in use Another service using the same port Stop conflicting service or change port in docker-compose.yml
Container exits immediately Command or entrypoint issue Check docker logs [container-name] for error messages

Rebuilding the Container

If you need to update dependencies or fix issues:

Force rebuild without cache:

make snaplogic-build-tools

Or rebuild with updated requirements:

make rebuild-tools

Checking Container Status

View running containers:

docker ps | grep tools

Check container logs:

docker logs snaplogic-test-example-tools-container

Debugging Tips

  • Enter the container: docker exec -it [container-name] bash
  • Check environment variables: docker exec [container-name] env
  • Inspect container details: docker inspect [container-name]
  • View network details: docker network inspect snaplogicnet
  • Check disk usage: docker system df
  • Clean up unused resources: docker system prune -a

Getting Help

If you encounter issues not covered here:

  1. Check the container logs for specific error messages
  2. Verify your .env file has all required variables
  3. Ensure Docker Desktop is running with sufficient resources
  4. Consult the project README for additional documentation
  5. Contact the team via Slack or open a GitHub issue

Docker Compose Profiles

Available profiles for selective service startup:

  • tools - Robot Framework test execution container
  • groundplex - SnapLogic Groundplex runtime
  • oracle-dev - Oracle database service
  • postgres-dev - PostgreSQL database service
  • mysql-dev - MySQL database service
  • kafka-dev - Kafka messaging service
  • minio-dev - MinIO S3-compatible storage

Useful Docker Commands

# View all containers (including stopped)
docker ps -a

# Remove all stopped containers
docker container prune

# View Docker disk usage
docker system df

# Clean up everything (containers, images, networks)
docker system prune -a --volumes

# Follow container logs in real-time
docker logs -f [container-name]

# Copy files from container
docker cp [container-name]:/path/to/file ./local/path

# Execute command in running container
docker exec [container-name] [command]
