🚀 Kafka Docker Compose Configuration

Complete documentation for SnapLogic Kafka KRaft deployment

Overview Services Configuration Networking Volumes Usage Troubleshooting

Overview

This Docker Compose configuration provides a complete Apache Kafka setup using KRaft mode (Kafka without Zookeeper). It includes the Kafka broker, a web-based UI for management, and utility services for setup, testing, and cleanup operations.

ℹ️
KRaft Mode: This configuration uses Kafka's KRaft consensus protocol, eliminating the need for Apache Zookeeper. This simplifies deployment and reduces resource overhead.

Key Features

  • ✅ KRaft mode (no Zookeeper dependency)
  • ✅ Web-based UI for monitoring and management
  • ✅ Health checks for service reliability
  • ✅ Resource limits for CI/CD stability
  • ✅ Multiple Docker profiles for different scenarios
  • ✅ Auto-topic creation enabled
  • ✅ Configurable ports via environment variables

Services

📊
Kafka Broker

The main Kafka broker service running in KRaft mode. This is the core message broker that handles all event streaming operations.

Environment Configuration

KAFKA_CFG_NODE_ID 1

Unique identifier for this Kafka node in the cluster

KAFKA_CFG_PROCESS_ROLES broker,controller

Node acts as both broker and controller in KRaft mode

KAFKA_CFG_LISTENERS PLAINTEXT://:29092, CONTROLLER://:9093, PLAINTEXT_HOST://:9092

Network listeners for different connection types

KAFKA_CFG_AUTO_CREATE_TOPICS_ENABLE true

Automatically create topics when first message is published

KAFKA_CFG_NUM_PARTITIONS 3

Default number of partitions for new topics

KAFKA_CFG_LOG_RETENTION_HOURS 168 (7 days)

How long to keep messages before deletion

KAFKA_HEAP_OPTS -Xmx512m -Xms256m

JVM heap memory configuration

KAFKA_KRAFT_CLUSTER_ID MkU3OEVBNTcwNTJENDM2Qk

Unique cluster identifier for KRaft mode

Resource Limits

  • Memory: 768M limit, 256M reservation
  • CPU: 1.0 CPU limit, 0.5 CPU reservation
🖥️
Kafka UI

Web-based user interface for Kafka cluster management and monitoring. Provides visual tools for topic management, consumer groups, and message browsing.

Configuration

KAFKA_CLUSTERS_0_NAME snaplogic-kafka-kraft

Display name for the Kafka cluster in UI

KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS kafka:29092

Kafka broker connection endpoint

DYNAMIC_CONFIG_ENABLED true

Allow runtime configuration changes

⚙️
Kafka Setup

One-time setup service that initializes topics and configurations after Kafka starts. Runs the /scripts/kafka-setup.sh script.

Profile: kafka
📤
Kafka Test Producer

Optional service for testing message production. Useful for validating Kafka connectivity and topic creation.

Profile: kafka-test
🧹
Kafka Cleanup

Utility service for cleaning up topics and resetting Kafka state. Use with caution as it may delete data.

Profile: kafka-cleanup

Docker Profiles

kafka
kafka-dev
kafka-test
kafka-cleanup

Health Checks

Services include health checks to ensure proper startup and operation:

  • Kafka: Verifies broker availability by listing topics
  • Kafka UI: Checks HTTP endpoint availability
  • Health checks include configurable intervals, timeouts, and retry counts
  • Start period allows services time to initialize before health checking begins

Networking

All services are connected through a custom bridge network named snaplogicnet. This provides:

  • Service discovery by container name
  • Isolated network environment
  • Internal communication between services
  • Controlled external access through mapped ports
💡
Listener Configuration: Kafka uses multiple listeners for different connection types. Internal services connect via kafka:29092, while external clients use localhost:9092.

Volumes

Persistent Storage

kafka-kraft-data /bitnami/kafka

Stores Kafka logs, metadata, and KRaft consensus data. Uses local driver for temporary storage in CI environments.

Script Mounts

./scripts/kafka /scripts:ro

Read-only mount for utility scripts used by setup, test, and cleanup services.

Usage

Starting Services

Start Kafka with the UI (development mode):

docker-compose -f docker-compose.kafka.yml --profile kafka-dev up -d

Start Kafka with setup scripts:

docker-compose -f docker-compose.kafka.yml --profile kafka up -d

Run test producer:

docker-compose -f docker-compose.kafka.yml --profile kafka-test up kafka-test-producer

Clean up Kafka data:

docker-compose -f docker-compose.kafka.yml --profile kafka-cleanup up kafka-cleanup

Accessing Services

  • Kafka UI: http://localhost:8080
  • Kafka Broker: localhost:9092 (for client connections)

Environment Variables

You can customize ports by setting environment variables:

export KAFKA_PORT=19092
export KAFKA_UI_PORT=18080
docker-compose -f docker-compose.kafka.yml --profile kafka-dev up -d

Troubleshooting

Common Issues

⚠️
Port Already in Use: If ports 9092 or 8080 are already occupied, set custom ports using environment variables as shown in the Usage section.

Checking Service Health

docker-compose -f docker-compose.kafka.yml ps

Viewing Logs

docker-compose -f docker-compose.kafka.yml logs -f kafka

Testing Kafka Connection

docker exec snaplogic-kafka-kraft kafka-topics.sh --bootstrap-server localhost:9092 --list

Creating a Test Topic

docker exec snaplogic-kafka-kraft kafka-topics.sh \ --bootstrap-server localhost:9092 \ --create --topic test-topic \ --partitions 3 --replication-factor 1

Resource Issues

If services fail to start due to resource constraints:

  • Check available Docker resources: docker system df
  • Clean up unused resources: docker system prune -a
  • Increase Docker Desktop memory allocation if needed
  • Consider reducing resource limits in the deploy section
