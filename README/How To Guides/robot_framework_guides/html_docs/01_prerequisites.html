1. System Requirements

Before getting started, ensure your system meets the following requirements:

Requirement Version Status Notes
Docker Desktop Latest stable Required Includes Docker Engine + Docker Compose
Git 2.x+ Required For cloning the repository
Make (GNU Make) 3.x+ Required Pre-installed on macOS/Linux; install on Windows via WSL or chocolatey
Python 3.12+ Recommended Only needed for local development; tests run inside Docker
VS Code Latest Recommended Best IDE for Robot Framework development
SnapLogic Account N/A Required Org admin access with API credentials

2. Tools & Knowledge

Familiarity with the following concepts will help you get the most from this framework:

Docker & Docker Compose

  • Container lifecycle (start, stop, logs)
  • Docker Compose profiles and services
  • Volume mounts and networking
  • Environment variable injection

Makefiles

  • Running make targets (make <target>)
  • Passing parameters (make target VAR=value)
  • Understanding target dependencies
  • Reading Makefile help output

Git Version Control

  • Clone, branch, commit, push
  • Understanding .gitignore (protects secrets)
  • Working with remote repositories

Robot Framework Basics

  • Keyword-driven test syntax
  • Variables: ${scalar}, @{list}, &{dict}
  • Tags for test selection
  • Settings, Test Cases, Keywords sections
New to Robot Framework? Visit robotframework.org for tutorials and documentation. The keyword-driven approach makes tests highly readable even for non-programmers.

3. Installation Steps

Step 1: Install Docker Desktop

Download and install Docker Desktop for your operating system:

# macOS (via Homebrew)
brew install --cask docker

# Or download from https://www.docker.com/products/docker-desktop

# Verify installation
docker --version
docker compose version

Step 2: Clone the Repository

# Clone the repository
git clone <repository-url> snaplogic-robotframework-examples
cd snaplogic-robotframework-examples

Step 3: Generate Your Project

# Generate a new project from the cookiecutter template
make generate-project

# Navigate into the generated project
eval $(make change-dir)

# Or manually:
cd <your_pipeline_name>

Step 4: Configure Environment

# Copy the example environment file
cp .env.example .env

# Edit with your SnapLogic credentials
vi .env   # or use your preferred editor
Important: The .env file contains sensitive credentials and is excluded from Git via .gitignore. Never commit this file to version control.

4. Pipeline Preparation

Before automating tests, you must have a working SnapLogic pipeline. Follow these pre-automation steps:

Pre-Automation Checklist

  1. Build your pipeline in the SnapLogic Designer UI
  2. Test it manually to confirm it works end-to-end
  3. Use Pipeline Parameters for account references (enables runtime flexibility)
  4. Export the pipeline as a .slp file from SnapLogic
  5. Save the .slp file to src/pipelines/ in your project

Why Pipeline Parameters Matter

Using pipeline parameters instead of hardcoded account references allows the framework to override them at runtime. This means the same pipeline can run against different database instances.

# Example: Instead of hardcoding a Snowflake account path,
# use a pipeline parameter like:
snowflake_acct = ../shared/snowflake_account

# The framework can then override at task execution time:
test_input_file = test_data.json

Exporting Pipelines from SnapLogic

  1. Open your pipeline in SnapLogic Designer
  2. Click the three-dot menu (⋮) on the pipeline
  3. Select Export
  4. Save the .slp file to your project's src/pipelines/ directory

5. Project Generation

This repository uses Cookiecutter to generate customized test projects. When you run make generate-project, it asks for:

Prompt Description Example
primary_pipeline_name Name for your test project (becomes the folder name) snowflake_etl_tests
included_systems Which database/service systems to include oracle,postgres,snowflake
What Gets Generated: A complete test project with Makefiles, Docker configurations, environment templates, test directories, and example test files tailored to your selected systems.

6. Verification Checklist

Run these commands to verify everything is set up correctly:

# 1. Verify Docker is running
docker info

# 2. Verify you're in the generated project directory
ls Makefile docker-compose.yml .env

# 3. Verify environment is configured
make check-env

# 4. Start services to confirm Docker setup works
make start-services

# 5. Check service status
make status
Ready to Go! If all checks pass, proceed to Architecture Overview to understand how the framework components work together.
