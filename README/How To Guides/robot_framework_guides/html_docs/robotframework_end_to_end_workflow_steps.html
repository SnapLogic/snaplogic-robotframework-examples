Framework Setup Requirements Setup Guide

This guide walks you through the prerequisites and setup needed before starting automation with SnapLogic Robot Framework. We follow a systematic approach to ensure smooth test execution.

Variable Convention: All variables in CAPITAL LETTERS are environment variables loaded from .env files. Lowercase variables like ${pipeline_name} are user-defined.

Step 1: Pipeline Preparation

Build and test your pipeline in SnapLogic Designer. Ensure it executes successfully and uses pipeline parameters for account names, database details, file paths, and any configurable values.

Why use pipeline parameters? Provides flexibility to execute pipelines with different data sets, allows runtime updates through triggered tasks, and makes pipelines reusable across environments.

Expected Outcome:

  • Valid .slp pipeline file ready for import
  • Clear understanding of required accounts
  • Knowledge of database/service credentials needed
  • Defined import locations for pipelines and accounts
Upload Pipeline To
/snaplogic-robotframework-examples/src/pipelines/

Step 2: Groundplex Configuration (Optional)

Option A: Use Existing Groundplex

If using a Groundplex already deployed in your SnapLogic environment, you only need the GROUNDPLEX_NAME in your .env file.

Option B: Deploy New Groundplex

If deploying a new Groundplex via automation, configure: GROUNDPLEX_NAME, GROUNDPLEX_ENV, RELEASE_BUILD_VERSION, and GROUNDPLEX_LOCATION_PATH.

VariableOption A (Existing)Option B (New)

GROUNDPLEX_LOCATION_PATHNot neededRequired
GROUNDPLEX_ENVNot neededRequired
RELEASE_BUILD_VERSIONNot neededRequired

Step 3: Account Configuration

Account templates and configurations are available at /snaplogic-robotframework-examples/env_files/. This directory is continuously expanding to support additional account types.

Directory Structure
env_files/ ├── database_accounts/ │ ├── .env.oracle │ ├── .env.mysql │ ├── .env.postgres │ └── .env.sqlserver ├── external_accounts/ │ └── .env.snowflake ├── messaging_service_accounts/ │ ├── .env.jms │ └── .env.kafka └── mock_service_accounts/ ├── .env.email ├── .env.s3 └── .env.salesforce
Override precedence: The root .env file has the highest precedence and will override the same variable in account-specific files under env_files/. This is useful for switching between environments (dev, staging, prod) without modifying account-specific files.

Step 4: Test Configuration

Each test file (e.g., snowflake_demo.robot) has a *** Variables *** section to configure:

Robot Test Variables Example
*** Variables *** # Pipeline details ${pipeline_name} snowflake ${pipeline_file_name} snowflake.slp ${task_name} snowflake_Task # Task parameters (inputs for pipeline execution) &{task_params_set} ... snowflake_acct=../${ACCOUNT_LOCATION_PATH}/${SNOWFLAKE_ACCOUNT_NAME} ... schema_name=${SNOWFLAKE_SCHEMA} ... table_name=${SNOWFLAKE_SCHEMA}.LIFEEVENTSDATA

Best Practices

Good Practices

Parameterize everything — use pipeline parameters for all configurable values.

Use meaningful names — e.g. ${snowflake_account_ref} instead of ${acc}.

Commit: .env.example, test files, pipelines.

Avoid

Don't commit: .env (contains credentials), actual_output/.

Avoid hard-coded values like ../shared/acc1.

Pipeline Execution: 5-Step Quick Start Quick Start

Get from zero to running tests in five steps.

1

Install Docker Desktop

Download and install Docker Desktop for your OS, start it, and verify installation.

Verify Docker
$ docker --version Docker version 24.x.x, build xxxxxxx
Windows users: Follow the Windows WSL VS Code Setup Guide for WSL, Ubuntu, make commands, and VS Code integration on Windows.
2

Clone the Repository

Create a working directory and clone the GitHub repository.

Git Clone
$ git clone https://github.com/SnapLogic/snaplogic-robotframework-examples
3

Open Project (IDE or Terminal)

Download VS Code (or any preferred IDE) and open the project folder, or work directly from the terminal.

Change Directory
# Option A: Use make $ eval $(make change-dir) # Option B: Direct cd $ cd {{cookiecutter.primary_pipeline_name}}
4

Configure Environment

Create a .env file at project root level. Copy contents from .env.example and update values as per project requirements.

Create .env File
$ cp .env.example .env # Then edit .env with your actual SnapLogic credentials, # organization details, and project settings. # Make sure to delete all inline comments during copy.
Important: Before automating tests, you must have a working SnapLogic pipeline (.slp file).
5

Build and Execute

Build your test environment using make commands. Choose between framework-managed Groundplex or existing Groundplex.

Case 1: Framework-Managed Groundplex

# Build Docker containers $ make start-services # Run tests with project space creation $ make robot-run-all-tests TAGS="oracle" PROJECT_SPACE_SETUP=True # Later executions (project space exists) $ make robot-run-all-tests TAGS="oracle"

Case 2: Existing Groundplex

# Build Docker containers $ make start-services # Run tests with existing Groundplex $ make robot-run-tests-no-gp TAGS="snowflake_demo" PROJECT_SPACE_SETUP=True # Later executions $ make robot-run-tests-no-gp TAGS="snowflake_demo"
Tip: When you get the latest code, there might be changes to requirement files or compose files. Run make clean-start or make clean-start-tools to rebuild. Build process takes about 2-3 minutes.

What Happens After Execution

Services Started

  • Groundplex launched for pipeline execution
  • Oracle Database started
  • PostgreSQL Database started
  • MinIO (S3-compatible) started & configured

In SnapLogic Org

  • Accounts are created
  • ProjectSpace is created
  • Project is created
  • Pipeline is imported
  • Triggered task is created & executed
Test Reports: Generated at test/robot_output/. Includes report.html (summary), log.html (detailed logs), and output.xml (CI/CD integration). Open HTML files in any browser.

Test-Execution Flow Testing Phases

The testing lifecycle consists of three major phases that prepare the infrastructure, initialize services, and execute tests.

Complete Testing Lifecycle
COMPLETE TESTING LIFECYCLE PHASE 1 PHASE 2 PHASE 3 Infrastructure ──> Service ──> Integration Setup Initialization Testing (createplex) (start-services (run-tests) + groundplex) ~3-5 min ~2-5 min Varies Creates project Starts Docker Executes Robot space, Snaplex, containers, Framework tests, downloads config waits for health, verifies results, sets permissions generates reports
PHASE 1

Infrastructure Setup (createplex)

This phase prepares the SnapLogic platform infrastructure. Only runs when PROJECT_SPACE_SETUP=True.

1

What Happens

Initialize Robot Framework → Delete existing project space (clean slate) → Create new project space → Create project → Create Snaplex configuration → Download Snaplex config files for Groundplex container.

Error Recovery: If project space deletion fails due to active Snaplex nodes, the framework automatically stops the Groundplex container, kills the JCC process, waits 60 seconds for nodes to deregister, retries, and exits with error if retry also fails.
Trigger Command
$ make robot-run-all-tests TAGS="oracle" PROJECT_SPACE_SETUP=True # Internally runs: $ make robot-run-tests TAGS="createplex" PROJECT_SPACE_SETUP=True
PHASE 2

Service Initialization

Starts all required Docker containers and waits for them to be healthy.

1

What Happens

Pull latest Docker images → Start all required services (databases, messaging, mock services) → Wait for health checks → Launch Groundplex container → Poll JCC status (20 attempts, 10s intervals) → Set file permissions (CI only).

Note: If using robot-run-tests-no-gp, the Groundplex launch is skipped entirely. The framework assumes your Groundplex is already running.
PHASE 3

Integration Testing

The main test execution phase where Robot Framework tests run against the SnapLogic platform.

Test Execution Sequence
__init__.robot (Suite Setup) │ ├── Load environment variables from .env files ├── Validate required variables (URL, credentials, etc.) ├── Set up global variables (paths, settings) └── Authenticate with SnapLogic API │ ▼ Test Suite Execution (filtered by TAGS) │ ├── Create Accounts │ └── Create Account From Template (JSON + Jinja substitution) │ ├── Upload Test Data Files │ └── Upload files to SLDB (CSV, JSON, expression libraries) │ ├── Import Pipelines │ └── Import from .slp files (src/pipelines/) │ ├── Create Triggered Tasks │ └── Create with parameters, notifications │ ├── Execute Tasks │ └── Run Triggered Task With Parameters (pipeline runs on Groundplex) │ └── Verify Results ├── Query database tables (record counts, data integrity) ├── Export data to CSV ├── Compare actual vs expected CSV files └── Validate file contents (JSON, CSV) │ ▼ Generate Reports ├── report-{timestamp}.html (summary) ├── log-{timestamp}.html (detailed steps) └── output-{timestamp}.xml (machine-readable)

Setup Mode vs Verify Mode

Setup Mode (PROJECT_SPACE_SETUP=True)

Creates/recreates entire infrastructure. Deletes and rebuilds project space. Error resilience with auto-retry. Time: 3-5 min setup + test time.

Use for: First-time setup, CI/CD pipelines, clean environment testing.

Verify Mode (PROJECT_SPACE_SETUP=False)

Verifies existing infrastructure. Fails fast if project space missing. No teardown/rebuild. Time: 1-2 min setup + test time.

Use for: Development, debugging, quick iteration, stable environments.

CI/CD Integration

The framework includes built-in Travis CI integration. The Makefile-based approach means you can easily adapt to Jenkins, GitHub Actions, GitLab CI, or any CI system.

CI/CD Pipeline Flow
Travis CI Pipeline │ ├── Environment Setup │ ├── Language: Python 3.12 │ ├── Distribution: Ubuntu Jammy │ ├── Services: Docker │ └── Set permissions, configure environment │ ├── Test Execution │ ├── make start-services │ ├── make robot-run-all-tests TAGS="..." Best Practices

Quality-as-Code: This framework embeds testing directly into the development workflow. Quality isn't a separate phase — it's part of every step.

Test Design

Tag tests appropriately for selective execution. Use templates for data-driven tests. Parameterize pipelines for flexibility. Create ephemeral data — tests create and clean up their own data.

Execution Strategy

Check status first with make status. Use Make commands to handle Docker complexity. Start small with smoke tests first. Fail fast on critical failures.

Test Execution Flow 3 Phases

1. Test Suite Initialization

Load Environment Variables (.env) • Spin Up Docker Containers • JSON auto-detection • Groundplex, DBs

2. Test Execution

Create Account (From Template) → Import Pipeline (SLP Files) → Execute Task (With Parameters)

3. Validation & Reporting

Verify Results (Assertions) • Generate Report (HTML/XML Output) → SnapLogic Platform (Target System)

Environment Variable Flow Config

Variables flow through four levels of the system.

Developer Level

.env files • env_files/*.env

Container Level

Docker Compose env_file: .env, env_files/*.env

Robot Framework Level

%{VAR} / ${VAR} • Set Global Variable

CI/CD Level

Travis / Jenkins env injection → HTML/XML Reports • Slack Notify

Key Insight: The root .env file has the highest precedence and overrides any variable defined in env_files/ subdirectories.
