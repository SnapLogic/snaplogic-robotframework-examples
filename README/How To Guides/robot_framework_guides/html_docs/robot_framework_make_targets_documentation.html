1. Understanding Robot Framework Tags

Tags are a powerful feature in Robot Framework that allow you to categorize, filter, and selectively run test cases. Understanding tags is essential before using the make targets, as the TAGS parameter determines which tests will be executed.

What Are Tags?

  • Labels for Test Cases: Tags are simple text labels attached to test cases that help categorize and identify them
  • Filter Mechanism: Tags allow you to selectively include or exclude test cases during execution
  • No Pre-definition Required: Tags don't need to be declared anywhere - just use them directly in your test cases
  • Case Insensitive: Tags are case-insensitive (SMOKE, smoke, and Smoke are treated the same)

How Tags Are Defined in Test Cases

Tags are defined using the [Tags] setting within a test case:

*** Test Cases ***
Create Triggered Task
    [Documentation]    Creates a triggered task for pipeline execution
    [Tags]    snowflake_demo    task_creation    smoke
    [Template]    Create Triggered Task From Template
    ${unique_id}    ${PIPELINES_LOCATION_PATH}    ${pipeline_name}    ${task_name}

Using Tags with Make Commands

The TAGS parameter in make commands filters which tests to run:

# Run only tests with 'snowflake_demo' tag
make robot-run-all-tests TAGS="snowflake_demo"

# Run tests with 'smoke' tag
make robot-run-all-tests TAGS="smoke"

# Run tests with multiple tags (OR logic - runs if ANY tag matches)
make robot-run-all-tests TAGS="snowflake_demo OR postgres_demo"

# Run tests with multiple tags (AND logic - runs only if ALL tags match)
make robot-run-all-tests TAGS="snowflake_demo AND task_creation"

# Exclude tests with specific tag
make robot-run-all-tests TAGS="NOT cleanup"

Common Tag Naming Conventions

Tag Type Examples Purpose
Feature Tags snowflake_demo, postgres_demo, oracle Identify tests for specific features or database integrations
Action Tags task_creation, task_execution, account Categorize by the type of operation being tested
Priority Tags smoke, regression, critical Indicate test importance or execution frequency
Setup Tags setup, cleanup, teardown, createplex Mark setup and cleanup test cases
Verification Tags verify_project_space_exists Tests that verify environment state

Special Tags Used by This Framework

Tag Used By Purpose
createplex robot-run-all-tests Creates project space and Snaplex when PROJECT_SPACE_SETUP=True
verify_project_space_exists Both targets Verifies project space exists or creates it (based on PROJECT_SPACE_SETUP)
Important: When running tests, always specify the appropriate TAGS to run only the tests you need. Running without TAGS may execute all tests in the suite, which could take significant time.

2. robot-run-all-tests

End-to-End Robot Test Workflow including full environment setup with Groundplex.

Usage

make robot-run-all-tests TAGS="your_tags" PROJECT_SPACE_SETUP=True|False

Execution Flow

Phase 1: Project Space → Phase 2: Launch Groundplex → Phase 2.1: Travis Permissions → Phase 3: Run Tests

Behavior Based on PROJECT_SPACE_SETUP

Setting Phase Behavior
True Phase 1 Create Project Space & Plex Runs: robot-run-tests TAGS="createplex" PROJECT_SPACE_SETUP=True
  • Deletes existing project space (if exists)
  • Creates new project space
  • Creates Snaplex (plex) inside project space
Phase 1 Error Handling If createplex fails due to "active Snaplex nodes":
  • Runs stop-groundplex to kill JCC and stop container
  • Waits 60 seconds for nodes to deregister
  • Retries createplex test
  • Exits with error if retry fails
Phase 2 Launch Groundplex Runs: launch-groundplex
  • Starts Docker containers using COMPOSE_PROFILES
  • Launches the Groundplex container
Phase 2.1 Travis Permissions Only on Travis CI: Sets chmod 777 on test data directories so Docker containers can read/write files during test execution.
Phase 3 Run Tests Runs: robot-run-tests TAGS="$(TAGS)" PROJECT_SPACE_SETUP=False
  • Executes user-specified tests with the provided TAGS
Use Case: Fresh environment setup, CI/CD pipelines, first-time setup
False
(Default)		 Phase 1 Verify Project Space Runs: robot-run-tests TAGS="verify_project_space_exists" PROJECT_SPACE_SETUP=False
  • Skips project space creation
  • Only verifies that project space already exists
  • Exits with error if project space doesn't exist
Phase 2 Launch Groundplex Runs: launch-groundplex
  • Starts Docker containers using COMPOSE_PROFILES
Phase 2.1 Travis Permissions Only on Travis CI: Sets chmod 777 on test data directories so Docker containers can read/write files during test execution.
Phase 3 Run Tests Runs: robot-run-tests TAGS="$(TAGS)" PROJECT_SPACE_SETUP=False
Use Case: Subsequent test runs, when project space already exists
Note: This target always launches Groundplex in Phase 2, regardless of PROJECT_SPACE_SETUP value.

3. robot-run-tests-no-gp

Robot Test Workflow WITHOUT Groundplex Launch. Use when Groundplex is already running or not needed.

Usage

make robot-run-tests-no-gp TAGS="your_tags" PROJECT_SPACE_SETUP=True|False

Execution Flow

Phase 1: Project Space → Phase 2: SKIPPED → Phase 2.1: Travis Permissions → Phase 3: Run Tests

Behavior Based on PROJECT_SPACE_SETUP

Setting Phase Behavior
True Phase 1 Create Project Space Only Runs: robot-run-tests TAGS="verify_project_space_exists" PROJECT_SPACE_SETUP=True
  • Deletes existing project space (if exists)
  • Creates new project space
  • Does NOT create Snaplex/Plex
Phase 1 Error Handling If fails due to "active Snaplex nodes":
  • Logs warning about active nodes
  • Does NOT attempt to stop Groundplex
  • User must manually resolve
Phase 2 Groundplex SKIPPED
  • No Groundplex/Plex operations performed
  • Assumes Groundplex is already running if needed
Phase 2.1 Travis Permissions Only on Travis CI: Sets chmod 777 on test data directories so Docker containers can read/write files during test execution.
Phase 3 Run Tests Runs: robot-run-tests TAGS="$(TAGS)" PROJECT_SPACE_SETUP=False
False
(Default)		 Phase 1 Verify Project Space Runs: robot-run-tests TAGS="verify_project_space_exists" PROJECT_SPACE_SETUP=False
  • Skips project space creation
  • Only verifies that project space exists
  • Exits with error if project space doesn't exist
Phase 2 Groundplex SKIPPED
Phase 2.1 Travis Permissions Only on Travis CI: Sets chmod 777 on test data directories so Docker containers can read/write files during test execution.
Phase 3 Run Tests Runs: robot-run-tests TAGS="$(TAGS)" PROJECT_SPACE_SETUP=False
Warning: This target does NOT launch Groundplex. Ensure Groundplex is already running if your tests require it.

5. When to Use Which Target

Scenario Recommended Target PROJECT_SPACE_SETUP
First time setup / Fresh environment robot-run-all-tests True
CI/CD pipeline - full workflow robot-run-all-tests True
Running tests after initial setup robot-run-all-tests False
Groundplex already running externally robot-run-tests-no-gp False
Tests don't need Groundplex robot-run-tests-no-gp False
Quick test iteration (GP already up) robot-run-tests-no-gp False
Create project space only (no plex needed) robot-run-tests-no-gp True
Reset project space with new plex robot-run-all-tests True

Quick Reference Commands

# Full setup with new project space and plex
make robot-run-all-tests TAGS="snowflake_demo" PROJECT_SPACE_SETUP=True

# Run tests using existing project space (launch GP)
make robot-run-all-tests TAGS="snowflake_demo"

# Run tests when GP is already running
make robot-run-tests-no-gp TAGS="snowflake_demo"

# Create project space only (no plex, no GP launch)
make robot-run-tests-no-gp TAGS="oracle" PROJECT_SPACE_SETUP=True

5. Technical Note: robot-run-tests (Base Target)

robot-run-tests is the underlying base target that both robot-run-all-tests and robot-run-tests-no-gp call internally to execute Robot Framework tests.

Usage (Direct)

make robot-run-tests TAGS="your_tags" PROJECT_SPACE_SETUP=True|False

What It Does

Step Description
1. Check Environment Runs check-env to verify .env file exists
2. Build Include Tags Converts TAGS parameter to Robot Framework --include arguments
3. Execute Tests Runs Robot Framework inside Docker tools container with:
  • -G $(DATE) - Group tests by date
  • --timestampoutputs - Timestamp output files
  • --variable PROJECT_SPACE_SETUP - Pass setup flag
  • --outputdir robot_output - Output directory
4. Report Result Prints success/failure message and exits with appropriate code
Recommendation: For standard workflows, use the higher-level targets:
  • robot-run-all-tests - Full workflow with Groundplex management
  • robot-run-tests-no-gp - Workflow without Groundplex operations
Use robot-run-tests directly only when you need fine-grained control over test execution without any environment setup phases.
