1. The 5-Step Quick Start

1 Install Docker Desktop 
# Download from https://www.docker.com/products/docker-desktop
# Verify installation:
docker --version
docker compose version
2 Clone & Generate Project 
git clone <repository-url>
cd snaplogic-robotframework-examples
make generate-project
3 Navigate to Generated Project 
# Option A: Auto-navigate
eval $(make change-dir)

# Option B: Manual
cd your_pipeline_name
4 Configure Environment 
cp .env.example .env

# Edit .env with your SnapLogic credentials:
# - URL (SnapLogic instance URL)
# - ORG_ADMIN_USER (your admin email)
# - ORG_ADMIN_PASSWORD (your password)
# - ORG_NAME (your organization name)
vi .env
5 Start Services & Run Tests 
# Start Docker services
make start-services

# Run tests — idempotent project-space setup happens automatically
# (creates project space/project if missing, reuses if already there)
make robot-run-all-tests TAGS="oracle"

# Verify-only / fast-fail mode (skips Snaplex re-registration):
make robot-run-all-tests TAGS="oracle" PROJECT_SPACE_SETUP=False
That's it! Your tests are now running. Results will be available in test/robot_output/ as HTML reports.

2. Two Execution Paths

Path A: Framework-Managed Groundplex (Recommended)

The framework creates and manages the Groundplex for you.

# Idempotent setup runs by default — creates project space/project
# if missing, reuses if already there. Never deletes existing assets.
make start-services
make robot-run-all-tests TAGS="oracle"

# Verify-only mode (skip Snaplex re-registration on quick re-runs):
make robot-run-all-tests \
  TAGS="oracle" \
  PROJECT_SPACE_SETUP=False
  • Idempotently ensures project space and Snaplex exist
  • Launches Groundplex container
  • Runs tests
  • Best for: CI/CD, first-time setup, repeat runs

Path B: Existing Groundplex

Use when you already have a running Groundplex.

# Idempotent project-space setup happens automatically.
# This target NEVER creates, starts, or stops a Groundplex —
# make sure yours is already running.
make start-services
make robot-run-tests-no-gp TAGS="oracle"

# Verify-only mode:
make robot-run-tests-no-gp \
  TAGS="oracle" \
  PROJECT_SPACE_SETUP=False
  • Idempotently ensures project space and project exist
  • Skips Snaplex registration and Groundplex creation/launch
  • Faster execution (no ~60s Groundplex startup)
  • Assumes a Groundplex is already running and attached to the project space
  • Best for: development, quick iteration, customer/CI environments with pre-provisioned Groundplex

What is "Verify-only mode"?

When you pass PROJECT_SPACE_SETUP=False, the framework switches from "create-if-missing" to "verify-it-exists." It does not attempt any creation — it just checks the project space is there and either passes or fails fast with an actionable error.

AspectDefault mode (=True)Verify-only mode (=False)
Create project space if missing ✅ Yes ❌ No — fails fast with the make commands you can run to create it
Create project if missing ✅ Yes ❌ No
Register Snaplex definition (REST call) ✅ Every run (idempotent) ❌ Skipped
Download .slpropz Groundplex config ✅ Every run (overwrites local) ❌ Skipped (assumes existing local copy)
Run your test tags ✅ Yes ✅ Yes (after the verify check passes)
When verify-only mode is the right choice:
  • Shared SnapLogic org without create permissions — your service account can't create project spaces; an admin set it up for you. Use verify-only so the framework doesn't try (and fail with a 403).
  • CI smoke checks — a separate setup job runs nightly; the test job just wants to confirm the environment is still in place before running.
  • Local quick iterations — you've already done the setup once, the Snaplex is registered, the .slpropz is on disk, and you want to skip the API round-trips for a faster inner loop.
  • You manually edited the local .slpropz for debugging (e.g., tweaking log levels) and don't want default mode to overwrite it.
What happens if the project space is missing in verify-only mode? The test fails with a multi-line console message that prints the exact make commands you can copy-paste to create it. Example: 
❌ Project space 'demo3_project_space' is not created in org 'ml-legacy-migration'.
============================================================

💡 To create the project space and project, run ONE of these:

  1) Full setup (creates project space + project + Snaplex, launches Groundplex):
      make robot-run-all-tests TAGS="<your-tags>"
      Example: make robot-run-all-tests TAGS="oracle"

  2) Without Groundplex (use when a Groundplex is already running externally):
      make robot-run-tests-no-gp TAGS="<your-tags>"
      Example: make robot-run-tests-no-gp TAGS="oracle"

ℹ️ PROJECT_SPACE_SETUP defaults to True (safe, idempotent — creates only what's missing).
   You explicitly passed PROJECT_SPACE_SETUP=False, which only VERIFIES the space exists.
   Drop the flag (or omit it) to let the framework create the project space for you.
============================================================

PROJECT_SPACE_SETUP Parameter

ValueBehaviorWhen to Use
True (default) Idempotent, non-destructive. Creates project space + project if missing, reuses if already there. Registers Snaplex (idempotent) and downloads .slpropz. Never deletes anything. Most cases — first-time setup, repeat runs, CI/CD. You don't need to pass it; it's the default.
False Verify-only. Asserts project space exists; fails fast with actionable guidance if not. Skips Snaplex re-registration and .slpropz re-download. Shared orgs without create permissions; CI smoke tests; quick iterations.

FORCE_RECREATE_PROJECT_SPACE Parameter (DESTRUCTIVE — opt-in)

⚠️ DANGER: Deletes the ENTIRE project space and every project, pipeline, account, and task inside it. Only use on a dedicated CI/regression project space you own exclusively.
ValueBehavior
True Deletes the whole project space and recreates it from scratch. Requires interactive confirmation (type the project space name) unless FORCE_CONFIRM=yes is also set.
False (default) No-op — respects the safe PROJECT_SPACE_SETUP behavior above.

4. Test Tags for Selective Execution

Tags control which tests run. Use them to execute only the tests you need:

# Run Oracle tests only
make robot-run-all-tests TAGS="oracle"

# Run Snowflake demo tests
make robot-run-all-tests TAGS="snowflake_demo"

# Run smoke tests (quick validation)
make robot-run-all-tests TAGS="smoke"

# Match multiple tags with OR (comma-separated, no spaces) —
# runs every test tagged with ANY of these:
make robot-run-all-tests TAGS="oracle,snowflake_demo,minio"
Note on tag syntax: The Makefile's TAGS variable is split on whitespace by Make's foreach, so the Robot Framework AND/OR/NOT keyword expressions (which require spaces) do not work here. Use comma-separated values for OR semantics. For more advanced filtering (AND/NOT/glob patterns), call robot directly inside the tools container with a single --include flag.

5. Viewing Test Results

After test execution, reports are generated in test/robot_output/:

FilePurposeBest For
report-*.html Summary report with pass/fail statistics Quick overview of results
log-*.html Detailed execution log with step-by-step output Debugging failures, understanding flow
output-*.xml Machine-readable XML output CI/CD integration, automated processing 
# Open the latest report in your browser (macOS)
open test/robot_output/report-*.html

# Or find the exact filename
ls -la test/robot_output/
