Problem Statement Critical Context

SnapLogic pipelines read from and write to Snowflake DB using JDBC/ODBC connectors. To build an automated testing framework (Robot Framework), we need a way to emulate or mock Snowflake locally so that tests can run in CI/CD without requiring a live Snowflake cloud account.

Key Constraint: SnapLogic connectors use JDBC/ODBC to talk to Snowflake — not Python APIs. Any emulator must expose a network endpoint that the Snowflake JDBC driver can connect to. Pure Python patching (in-process) will NOT work for SnapLogic.

Why Not Just Use Postgres?

While Postgres and Snowflake both use SQL-like syntax, Snowflake has its own dialect ("Snowflake SQL") with specific language extensions, semi-structured data types (VARIANT, ARRAY, OBJECT), and functions that Postgres cannot parse. A Postgres database cannot reliably test a Snowflake job.

LocalStack for Snowflake Best Overall

A commercial Docker-based Snowflake emulator from LocalStack (the same company behind the popular AWS emulator). Currently at version 1.7+ as of late 2025.

How it works: Runs a Docker container (localstack/snowflake) that exposes a Snowflake-compatible endpoint at snowflake.localhost.localstack.cloud. Point your existing Snowflake connectors (Python, JDBC, ODBC, Snowpark, dbt) to this local hostname. Everything runs locally with zero cloud calls.

Supported Features

  • Warehouses, Databases, Schemas, Tables
  • User/Data/Named Stages
  • COPY INTO operations
  • Materialized Views & Dynamic Tables
  • Window Functions
  • VARIANT operations & semi-structured data
  • Snowflake Scripting
  • Information_schema views
  • Aggregate and Bitwise functions
  • Polaris Catalog (Iceberg tables)
  • JDBC, ODBC, Snowpark, dbt, DBeaver, Flyway, Streamlit

Not Supported

  • Time Travel
  • Zero-copy Cloning
  • Streamlit in Snowflake
  • Advanced data-sharing capabilities

✅ Pros

  • Highest Snowflake SQL compatibility
  • Native JDBC/ODBC support (works with SnapLogic)
  • Docker-based — CI/CD ready
  • Actively maintained (v1.7+)
  • Supports dbt workflows

❌ Cons

  • No free tier (only free trial)
  • Requires LOCALSTACK_AUTH_TOKEN
  • Some edge-case Snowflake behaviors may differ

fakesnow (by tekumara) Best Free Option

An open-source Python library and optional HTTP server that fakes the Snowflake Connector for Python, backed by DuckDB and SQLGlot for SQL translation.

Two Modes:
1. In-process patching: with fakesnow.patch(): intercepts snowflake.connector.connect() calls and routes to DuckDB
2. HTTP server: Runs a local HTTP endpoint for non-Python clients (tested with snowsql CLI)

Supported Features

  • Standard SQL operations and cursors
  • Information schema queries
  • Multiple databases
  • Parameter binding in queries
  • Table comments
  • Pandas integration (write_pandas(..))
  • Result batch retrieval (get_result_batches())
  • COPY INTO from S3/stages (with credential config)

Limitations (Detailed)

Critical for SnapLogic: SnapLogic uses JDBC/ODBC, but fakesnow's HTTP server has only been tested with snowsql CLI — not with Snowflake JDBC drivers. Compatibility with SnapLogic is unverified.

Python Usage
# Install pip install fakesnow pip install fakesnow[server] # for HTTP server mode # In-process patching (Python only) import fakesnow import snowflake.connector with fakesnow.patch(): conn = snowflake.connector.connect( user="fake", account="fake" ) cur = conn.cursor() cur.execute("CREATE TABLE test (id INT, name VARCHAR)") cur.execute("INSERT INTO test VALUES (1, 'hello')") cur.execute("SELECT * FROM test") print(cur.fetchall()) # HTTP Server mode (for non-Python clients) # $ fakesnow-server --port 8080

Go Snowflake Emulator (nnnkkk7) Free / Docker-Ready

A lightweight open-source Snowflake emulator written in Go, using DuckDB as the backend. Announced on Hacker News in early 2026.

How it works: Provides a Snowflake-compatible REST API (SQL API v2). Runs as a Docker container (ghcr.io/nnnkkk7/snowflake-emulator:latest) with in-memory or persistent storage options.

Features

  • Snowflake SQL API v2 compatible REST endpoint
  • Supports the gosnowflake driver
  • Basic DDL/DML and SELECT queries
  • Docker container ready
  • In-memory or persistent storage modes

Limitations

  • No authentication support
  • No clustering
  • No external stages
  • No JavaScript stored procedures
  • Very early-stage project (2026)
  • REST API only — JDBC compatibility unknown

✅ Pros

  • Free and open source
  • Docker-ready for CI/CD
  • Lightweight Go binary
  • Network endpoint (REST API)

❌ Cons

  • Very new — limited adoption
  • REST API only, no native JDBC
  • Limited SQL coverage
  • SnapLogic JDBC compatibility untested
Docker Quick Start
# Pull and run the Go Snowflake emulator docker pull ghcr.io/nnnkkk7/snowflake-emulator:latest docker run -d -p 8080:8080 ghcr.io/nnnkkk7/snowflake-emulator:latest # Connects via Snowflake SQL API v2 # Endpoint: http://localhost:8080
Also available: A Rust-based emulator by sivchari (built on Apache DataFusion) is also available at github.com/sivchari/snowflake-emulator. It is even more experimental but worth watching.

snowflake-vcrpy (Snowflake Labs) Record & Replay

An official Snowflake Labs pytest plugin that records and replays HTTP interactions from the Snowflake Python driver, built on VCR.py.

How it works: On the first test run, HTTP requests/responses are recorded to YAML "cassette" files. On subsequent runs, VCR.py intercepts matching HTTP requests and returns recorded responses. Zero Snowflake connection needed after initial recording.

How It Works

1

Record Phase (requires real SF)

Run tests against real Snowflake. HTTP interactions are saved to YAML cassette files under <test_folder>/cassettes/.

2

Replay Phase (fully offline)

VCR.py intercepts matching HTTP requests and returns pre-recorded responses. Tests run instantly with no Snowflake connection.

3

Re-record When Needed

When schemas or data change, re-record cassettes against real Snowflake to refresh responses.

Limitations

  • Requires initial real Snowflake connection for recording
  • Python connector only — does NOT work with JDBC/ODBC
  • Cassettes go stale when schema or data changes
  • Not suitable for testing new/changed queries without re-recording
  • Tests validate recorded behavior, not live behavior
SnapLogic Impact: Since this only works with the Python Snowflake driver, it is NOT directly usable for SnapLogic JDBC testing. However, it could be useful for supplementary Python-based validation tests alongside the main SnapLogic test suite.
Usage Example
import pytest import snowflake.connector # Annotate test with snowflake_vcr to enable record/replay @pytest.mark.snowflake_vcr def test_query_data(): conn = snowflake.connector.connect( user="user", account="account", password="password" ) cur = conn.cursor() cur.execute("SELECT * FROM my_table LIMIT 10") results = cur.fetchall() assert len(results) == 10

DuckDB + SQLGlot Transpilation Build-Your-Own

Use DuckDB as a local analytical database and SQLGlot to transpile Snowflake SQL into DuckDB-compatible SQL. About 90% of typical analytical SQL works without changes.

Architecture: Wrap DuckDB behind a thin service that accepts Snowflake SQL, transpiles it via SQLGlot, and executes on DuckDB locally. You would need to build a lightweight adapter/proxy yourself.

DuckDB Snowflake Extension

The DuckDB community provides a snowflake extension that adds compatibility for Snowflake-specific functions, bringing coverage to approximately 90% of analytical queries.

SQLGlot Executor (Alternative — No DB needed)

SQLGlot also includes a pure Python executor that can run Snowflake SQL directly against in-memory Python dictionaries. No database needed at all — ideal for unit testing small datasets.

✅ Pros

  • Completely free and open source
  • DuckDB is production-grade and fast
  • SQLGlot is battle-tested (used by fakesnow, UniverSQL)
  • ~90% analytical SQL compatibility
  • No network dependency

❌ Cons

  • Requires custom engineering to build the proxy/adapter
  • No VARIANT type natively in DuckDB
  • No Snowflake-specific features (stages, streams, tasks)
  • Not all Snowflake functions have DuckDB equivalents
  • SnapLogic JDBC integration requires building a JDBC-compatible wrapper
SQLGlot Transpilation Example
import sqlglot # Transpile Snowflake SQL to DuckDB snowflake_sql = """ SELECT PARSE_JSON(payload):customer_id::INT AS customer_id, DATEADD('day', -7, CURRENT_TIMESTAMP()) AS week_ago FROM raw_events WHERE event_type = 'purchase' QUALIFY ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY ts DESC) = 1 """ duckdb_sql = sqlglot.transpile( snowflake_sql, read="snowflake", write="duckdb" )[0] print(duckdb_sql) # SQLGlot Pure Executor (no DB needed) from sqlglot.executor import execute result = execute( "SELECT id, name FROM users WHERE active = TRUE", dialect="snowflake", tables={"users": [ {"id": 1, "name": "Alice", "active": True}, {"id": 2, "name": "Bob", "active": False}, ]} ) print(result) # [(1, 'Alice')]

Recommended Phased Approach Architecture Decision

Given that SnapLogic uses JDBC/ODBC to connect to Snowflake, here is the recommended phased strategy to unblock testing immediately while evaluating long-term local emulation.

1

Phase 1: Immediate Unblock (Now)

Start with a Snowflake Trial Account (free, 30 days, $400 credits). This gives 100% Snowflake compatibility with zero setup. Begin testing SnapLogic pipelines immediately while evaluating local options.

2

Phase 2: Evaluate Free Emulators (Week 1–2)

Test the Go Snowflake Emulator in Docker. It's free, exposes a REST API, and is the most promising free option with a network endpoint. Validate whether SnapLogic's JDBC driver can connect to its API.

3

Phase 3: Record Safety Net (Parallel)

While the trial is active, use snowflake-vcrpy to record real Snowflake HTTP responses. These cassettes serve as a safety net for Python-based validation tests even after the trial expires.

4

Phase 4: Fallback — LocalStack (If needed)

If the Go emulator does not meet JDBC requirements, request budget for LocalStack for Snowflake ($29/mo). It has proven JDBC/ODBC support and is the best long-term local emulation option.

Decision Matrix

Scenario Recommended Tool Why
Need to start testing NOW SF Trial Zero setup, 100% compatible, free 30 days
Budget available for best local option LocalStack JDBC native, highest SF SQL coverage, Docker CI/CD
Must be free + local + JDBC Go Emulator Best free candidate with network endpoint (needs validation)
Python-only testing (no JDBC needed) fakesnow Best free Python mock, DuckDB-backed, easy setup
Speed up existing Python tests snowflake-vcrpy Record once, replay forever, zero cloud cost after recording
SQL logic validation (no DB needed) DuckDB + SQLGlot Transpile SF SQL to DuckDB, ~90% coverage, free
Bottom Line: For SnapLogic pipeline testing specifically, the most pragmatic path is: SF Trial (now) → Evaluate Go Emulator (free) → LocalStack ($29/mo) if needed. This gives you zero downtime, a free exploration path, and a proven fallback.
