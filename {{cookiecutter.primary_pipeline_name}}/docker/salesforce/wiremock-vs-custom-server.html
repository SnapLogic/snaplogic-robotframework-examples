WireMock vs Custom Server

Architecture Comparison for Salesforce Mock API Testing

SnapLogic QA Team Salesforce Snap Pack Docker Compose REST + Bulk API SOQL Parsing

The Problem

Why do we need a mock Salesforce server?

Testing SnapLogic Salesforce Pipelines Requires a Salesforce Endpoint

  • Real Salesforce orgs have API rate limits and cost money
  • Shared orgs cause test data conflicts between team members
  • Test data in production orgs creates compliance risks
  • Network latency makes test suites slow
  • No control over the Salesforce environment (schema changes, downtime)
💡 Solution

Run a local mock server inside Docker that mimics the Salesforce API — fast, free, isolated, and fully controlled. From SnapLogic's perspective, it cannot tell the difference.

Two Approaches Compared

◉ WireMock + JSON Server

Static URL pattern matching with hardcoded responses. Each API operation needs a separate JSON mapping file.

Status: Original implementation

⚡ Custom Express.js Server

Dynamic route handlers with in-memory database. Schema-driven — one JSON file per Salesforce object.

Status: New implementation (replacement)

◉ WireMock Approach

Static URL pattern matching with hardcoded responses

📄 How It Works

Each API operation needs a separate JSON mapping file that says: "When you see this URL, return this response."

⚠️ File explosion

5 objects × 6 mappings = 30+ JSON files to maintain

⚠ Limitations

  • Static responses — Read always returns the same hardcoded record, even if you created different data
  • No validation — Missing required field "Name"? WireMock still returns success
  • No real SOQL — WHERE clause is ignored, always returns same data
  • Create & Read are disconnected — Create returns random Id, Read returns a different hardcoded Id
  • 2 containers needed — WireMock + JSON Server (separate networking)

⚡ Custom Server Approach

Dynamic route handlers with in-memory database

⚡ How It Works

ONE route handler function handles all objects dynamically using a :object URL wildcard.

rest-routes.js — Create Handler
// One handler for ALL objects
app.post('/sobjects/:object', (req, res) => {
  const schema = schemas[req.params.object];

  // Step 1: Validate against schema
  const errors = validate(req.body, schema);
  if (errors.length > 0) return res.json(errors);

  // Step 2: Generate Salesforce-style Id
  const id = generateId(schema.idPrefix);

  // Step 3: Store in memory
  database[objectName].push(record);

  // Step 4: Return real response
  res.json({ id, success: true });
});

✓ Minimal footprint

5 objects = 5 schema files. Zero code changes. Add a new object by creating one JSON file.

✓ Capabilities

  • Real CRUD — Create a record, Read returns the actual created data
  • Field validation — Required fields, picklist values, max length, read-only fields
  • Real SOQL — WHERE, LIKE, IN, ORDER BY, NULLS FIRST/LAST, LIMIT all work on actual data
  • Single container — Everything in one Express.js server
  • Schema-driven — Add new object = add one JSON file, zero code changes
  • Bulk API support — Both v1 (XML) and v2 (CSV) ingest and query

Read / SOQL Query — Step by Step

What happens when a SnapLogic SOQL snap sends a query

📄 WireMock

Query: SELECT Id, Name FROM Account WHERE Type = 'Customer'

WireMock mapping — ignores the actual SOQL
{
  "request": {
    "queryParameters": {
      "q": { "contains": "Account" }
    }
  },
  "response": {
    "jsonBody": {
      "records": [
        { "Id": "001TEST01",
          "Name": "Test Account" }
      ]
    }
  }
}

Always returns the same record regardless of what data was created or what WHERE clause is used. The query is only checked for containing "Account".

⚡ Custom Server

Same Query: SELECT Id, Name FROM Account WHERE Type = 'Customer'

Parses SOQL, filters actual data
const parsed = parseSOQL(query);
// → { fields: ['Id','Name'], object: 'Account',
//     where: [{field:'Type', op:'=', val:'Customer'}] }

let records = database['Account'];
records = applyWhere(records, parsed.where);
records = applyOrderBy(records, parsed.orderBy);
// → returns only actual Customer records

Returns real data that was actually created. WHERE, LIKE, IN, ORDER BY, NULLS FIRST/LAST, LIMIT all work against the in-memory database.

Adding a New Object

📄 WireMock: 6–8 steps

  • Create describe mapping file
  • Create create mapping file
  • Create read-query mapping file
  • Create read-single mapping file
  • Create update mapping file
  • Create delete mapping file
  • Add JSON Server endpoint config
  • Create response body files with hardcoded data

⚡ Custom Server: 2 steps

  • Create one schemas/ObjectName.json file
  • Restart the container
  • Done. All CRUD + SOQL + Bulk API work automatically.
