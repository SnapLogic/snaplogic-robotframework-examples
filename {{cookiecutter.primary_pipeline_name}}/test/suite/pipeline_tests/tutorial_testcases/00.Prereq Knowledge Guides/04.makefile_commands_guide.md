# Makefile Commands — A Layman's Guide

## TL;DR

A `Makefile` is just a **shortcut book**. Instead of typing long, complicated commands every time, you type `make <something>` and it runs the full command for you.

> **Think of Makefile commands as the "speed-dial buttons" on an old phone.**
> The number behind the button might be long and hard to remember, but you just press one button and it dials for you.

### First, a quick vocabulary note: what is a **target**?

Each shortcut inside a Makefile is called a **target**. When you type `make oracle-start`, the word `oracle-start` is the target name — it's just the **label** of that shortcut. The Makefile looks up the matching label and runs the recipe (the actual commands) stored under it.

So everywhere you see commands like `make help`, `make start-services`, or `make robot-run-all-tests`, the word after `make` is just a target name pointing at one specific recipe inside one of the Makefile files.

---

## Why does this repo use Makefiles?

Imagine starting an Oracle database container manually. The real command looks like this:

```bash
COMPOSE_PROFILES=oracle-dev docker compose \
  --env-file env_files/database_accounts/.env.oracle \
  --env-file env_files/database_accounts/.env.db2 \
  --env-file env_files/database_accounts/.env.mysql \
  --env-file .env \
  -f docker-compose.yml \
  up -d oracle-db
```

Would you remember that every time? Of course not.

So instead, someone wrote a Makefile shortcut:

```bash
make oracle-start
```

**Same thing happens. Two words instead of ten lines.** That's the whole idea.

## The pattern that makes everything easy

Once you learn the **pattern**, you already know dozens of commands. Every database/service follows the same shape:

```bash
make <service>-start      # Start the container
make <service>-stop       # Stop the container
make <service>-status     # See if it's running
make <service>-logs       # View logs (for debugging)
make <service>-load-data  # (databases only) Load test data
```

So once you know `oracle-start`, you automatically know:

- `postgres-start`, `mysql-start`, `sqlserver-start`, `kafka-start`, `minio-start`, `db2-start`...
- And all of them have matching `-stop`, `-status`, `-logs`.

**That's the power of consistency** — the Makefile recipes are written the same way across all services.

## The most useful commands for daily work

> You don't need to memorize all of them. Start with these 9 and you can do 90% of testing work.

| Command | What it does |
|---------|--------------|
| `make help` | **Shows you every available command in this repo** ← start here |
| `make start-services` | Start ALL services at once (databases + mocks + tools) |
| `make start-tools-service-only` | Start ONLY the test tools container (lighter, faster) |
| `make oracle-start` (or `postgres-start`, etc.) | Start one specific database |
| `make robot-run-all-tests TAGS="oracle"` | Run tests AND let the framework bring up a **local Docker Groundplex** for the tests to execute on |
| `make robot-run-tests-no-gp TAGS="oracle"` | Run tests but the framework **never touches the Groundplex** — you bring your own (local you started yourself, or external/cloud) |
| `make status` | See which containers are running right now |
| `make launch-groundplex` | Start the SnapLogic Groundplex container |
| `make clean-start` | Tear everything down and rebuild from scratch |

### `robot-run-all-tests` vs `robot-run-tests-no-gp` — the **real** difference

Both targets do **almost the same work**. The only meaningful difference is **who is responsible for the Groundplex**.

| What it does | `robot-run-all-tests` | `robot-run-tests-no-gp` |
|--------------|----------------------|-------------------------|
| Create / reuse project space | ✅ yes | ✅ yes (same) |
| Create / reuse project inside it | ✅ yes | ✅ yes (same) |
| Create accounts (Oracle, S3, etc.) | ✅ yes | ✅ yes (same) |
| **Bring up the Groundplex** | ✅ **framework launches a local Docker Groundplex** | ❌ **framework never touches the Groundplex** |
| Run your tests | ✅ yes | ✅ yes (same) |

> **The `-no-gp` suffix literally means "no Groundplex management."**
>
> It is **not** a "lighter / lesser" version of the other command. It still does all the project-space setup, account creation, and test execution. It just refuses to start, stop, or restart a Groundplex — because *you* (the user) are responsible for that one.

### So which one should I use?

Ask yourself: **"Where is the Groundplex that will execute my pipelines?"**

| Where the Groundplex lives | Use this | Why |
|-----------------------------|----------|-----|
| 🐳 **You want the framework to spin up a local Docker Groundplex for you** | `make robot-run-all-tests` | The framework starts a `snaplogic-groundplex` container and your tests execute on it. Easiest for local dev. |
| 🖥️ **You already have your own Groundplex running** (local Docker, your own VM, your laptop, a Plex you launched manually with `make launch-groundplex`) | `make robot-run-tests-no-gp` | You own the Groundplex lifecycle. Framework should not start/stop it on your behalf. |
| ☁️ **You're running against a cloud / customer-hosted / external Groundplex** | `make robot-run-tests-no-gp` | Same reason — the framework must stay hands-off. There's even a safety guard (`EXTERNAL_GROUNDPLEX=yes`) that **blocks** `robot-run-all-tests` from being used here, to prevent accidentally starting a duplicate local Plex with the same name. |

## How to discover commands without reading code

Two simple ways:

### 1. `make help`

```bash
make help
```

This prints **every available command** with a short description. It's the Makefile's built-in cheat sheet — way easier than opening files.

### 2. Browse the makefiles folder

```bash
ls makefiles/
ls makefiles/database_services/
```

Each `Makefile.<something>` is just a text file. Open it in any editor — the targets at the top of the file are the commands you can run. You don't need to understand the recipe details to know *what's available*.

## The big takeaway 💡

> **Makefile = a friendly shortcut layer that hides ugly long commands behind short, memorable names.**
>
> - You **don't need to know Docker syntax** to start Oracle — just `make oracle-start`.
> - You **don't need to remember env-file paths** to run tests — just `make robot-run-all-tests`.
> - The Makefile recipes are written **once** by the framework authors. You **reuse** them every day.
>
> If you find yourself typing the same long command twice in this repo, there's probably already a `make` target for it. Check `make help` first.
