## 1. Manual testing vs Automation — a layman's analogy

Before diving into what `__init__.robot` actually does, here's the most important mental model: **`__init__.robot` is doing exactly what you do manually before testing a pipeline — just automatically, every time.**

### What you do manually (before any automation existed)

When your team lead hands you a task — *"please test pipeline X"* — you don't just open SnapLogic and click around. You first **gather all the prerequisite information**:

```
┌─────────────────────────────────────────────────────────────┐
│  YOU (manual tester) — Day 1 of testing a new pipeline      │
└─────────────────────────────────────────────────────────────┘

  Step 1: Ask your lead 📋
    ┌──────────────────────────────────────────────────────┐
    │ "What org should I use?"          → ml-legacy-...    │
    │ "What's my username/password?"    → admin / ****     │
    │ "Which project space?"            → test9_space      │
    │ "Which project inside it?"        → demo_project     │
    │ "Which Groundplex runs it?"       → dev_plex         │
    │ "Which accounts do I need?"       → Oracle, S3, ...  │
    │ "Where do the pipelines live?"    → shared/ folder   │
    └──────────────────────────────────────────────────────┘
                          │
                          ▼
  Step 2: Open SnapLogic 🌐
    Log in → check if project space exists
              ├─ exists? → good, use it
              └─ missing? → create it manually
                            create project inside it
                            create the shared/ accounts folder

  Step 3: Create accounts 🔑
    Click "New Account" → fill Oracle host/user/password
    Click "New Account" → fill S3 keys
    ... repeat for every system the pipeline touches

  Step 4: Upload pipeline & run it 🚀
    Import the .slp file → execute → check results
```

That's a lot of clicking. And every tester does it the same way every time.

### What `__init__.robot` does (the automation)

The init file does **exactly the same thing**, just by reading instructions from a file instead of asking a human.

```
┌─────────────────────────────────────────────────────────────┐
│  ROBOT FRAMEWORK — automatically, every test run            │
└─────────────────────────────────────────────────────────────┘

  Step 1: Read the "lead's brief" 📋
    .env file = your team lead's instructions, in writing
    ┌──────────────────────────────────────────────────────┐
    │ ORG_NAME=ml-legacy-migration                         │
    │ ORG_ADMIN_USER=admin                                 │
    │ ORG_ADMIN_PASSWORD=****                              │
    │ PROJECT_SPACE=test9_space                            │
    │ PROJECT_NAME=demo_project                            │
    │ GROUNDPLEX_NAME=dev_plex                             │
    │ ORACLE_HOSTNAME=oracle-db ... (account info)         │
    └──────────────────────────────────────────────────────┘
                          │
                          ▼
  Step 2: Talk to SnapLogic API 🌐
    Auth using ORG_ADMIN_USER / PASSWORD
       → check if project space exists
              ├─ exists? → reuse it
              └─ missing? → create it via API
                            create project via API
                            create shared/ accounts folder

  Step 3: Create accounts 🔑
    For each .env file in env_files/, render the JSON template
       → POST to SnapLogic → account exists in shared/

  Step 4: Hand off to your test cases 🚀
    Your .robot test cases now run against a fully-prepared env
```

### Side-by-side: manual vs automated

| Manual step | What you do | Automated equivalent in `__init__.robot` |
|-------------|-------------|------------------------------------------|
| Ask the lead for org, creds, project info | Read an email / Slack message / ticket | **Load Environment Variables** — read `.env` files (Step 1) |
| Figure out *how* to log in | Username/password? Or SSO token? | **Detect Auth Method** — auto-pick basic/jwt/oauth2 (Step 2) |
| Sanity-check you have everything | "Wait, did the lead give me the password?" | **Validate Environment Variables** — fail fast if missing (Step 3) |
| Remember where pipelines / accounts live | Bookmark folders in SnapLogic UI | **Set Up Global Variables** — expose paths to all tests (Step 4) |
| Create project space if missing, otherwise reuse | Click around in SnapLogic Manager | **Project Set Up** — idempotent create via API (Step 5) |

### The key insight 💡

> **`.env` is to automation what a "lead's briefing email" is to manual testing.**
>
> It contains every piece of context a tester would need:
> - Which org and credentials to use
> - Which project space / project to work in
> - Which Groundplex to run pipelines on
> - Which accounts to set up (Oracle, S3, Snowflake, etc.)
>
> The `__init__.robot` file is the *automated* version of the tester who reads that briefing and does the prep work — every time, without forgetting a step.

### And the best part — it runs *once* regardless of test count

Whether you run **1 test case** or **100 test cases**, `__init__.robot` runs **exactly once** at the start of the suite. The prep work is shared across all tests:

```
                  ┌──────────────────┐
                  │  __init__.robot  │  ← runs ONCE
                  │  (the prep work) │
                  └────────┬─────────┘
                           │
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
   ┌─────────┐        ┌─────────┐        ┌─────────┐
   │ test 1  │        │ test 2  │  ...   │ test N  │
   └─────────┘        └─────────┘        └─────────┘
        └──────────────────┴──────────────────┘
              All share the same prepared env
```

This is why manual testers love automation: the prep work happens once, deterministically, and then dozens of tests can run against the same clean setup.

## 2. The big picture

Robot Framework looks for a file named `__init__.robot` in **every test-suite directory** when a suite is executed. The `Suite Setup` declared inside runs *once*, before any test case in any sub-directory.

> **File location:** `test/suite/__init__.robot`
> **Trigger:** automatic — Robot finds it; you never call it
> **Purpose:** load `.env` values, detect auth, create the SnapLogic project space + accounts before any pipeline test runs

### Execution Flow

```
▸ make robot-run-tests-no-gp TAGS="sqlserver2"     (user runs make)
        ↓
▸ Robot Framework discovers the test file and walks up directories
  looking for __init__.robot
        ↓
▸ __init__.robot Suite Setup → Before Suite keyword runs
        ↓
▸ 5 sub-keywords run sequentially (covered below)
        ↓
▸ Only after all 5 succeed, the actual test cases run
  (e.g. sql_server2.robot, oracle_baseline_tests.robot)
```

### The 5 sub-keywords inside `Before Suite`

```robot
Before Suite
    Load Environment Variables       # Step 1 — read .env files
    Detect Auth Method               # Step 2 — basic / jwt / oauth2 / sltoken
    Validate Environment Variables   # Step 3 — fail fast on missing vars
    Set Up Global Variables          # Step 4 — expose paths to all tests
    Project Set Up-...-Create Accounts  # Step 5 — talk to SnapLogic API
```

## 3. Step 1 — Load Environment Variables

This is the single most important step. It's where `.env` files become `${VARIABLE}` values that the rest of the test suite can use.

### Three layers, last layer wins

Three separate sources are loaded in a **specific order**. If the same variable is defined in multiple places, the **later layer overrides** the earlier one.

| Priority | Layer | Source |
|----------|-------|--------|
| LOWEST | **Layer 1** | All `.env*` files in `env_files/` and its sub-directories |
| MIDDLE | **Layer 2** | Root `.env` file at the project root (overrides Layer 1) |
| HIGHEST | **Layer 3** | File passed via `ENV=path/to/file` on the command line |

### What lives where

| Layer | Folder / File | What goes there |
|-------|---------------|-----------------|
| 1 | `env_files/database_accounts/.env.sqlserver` | SQL Server host, port, user, password |
| 1 | `env_files/database_accounts/.env.oracle` | Oracle defaults |
| 1 | `env_files/messaging_service_accounts/.env.kafka` | Kafka defaults |
| 1 | `env_files/mock_service_accounts/.env.s3` | MinIO / S3 mock |
| 2 | `.env` (project root) | SnapLogic org credentials, project space name, per-test overrides |
| 3 | e.g. `.env.stage` | Environment-specific overrides (passed via `ENV=.env.stage`) |

### Concrete example

Imagine you have:

```bash
# env_files/database_accounts/.env.sqlserver  (Layer 1)
SQLSERVER_DATABASE=master
SQLSERVER_HOST=sqlserver-db

# .env  (Layer 2 — overrides Layer 1)
SQLSERVER_DATABASE=TEST
ORG_NAME=ml-legacy-migration
PROJECT_SPACE=test9_project_space
```

Result after `Load Environment Variables`:

- `${SQLSERVER_DATABASE}` = `TEST` ✓ Layer 2 won
- `${SQLSERVER_HOST}` = `sqlserver-db` ✓ from Layer 1, no override
- `${ORG_NAME}`, `${PROJECT_SPACE}` = from Layer 2

## 4. Step 2 — Detect Auth Method

SnapLogic supports four authentication methods. The framework picks one automatically unless you set `AUTH_METHOD` explicitly.

### Decision tree

```
Is ${AUTH_METHOD} set in .env?
├─ YES → use that value
└─ NO  → auto-detect:
         ├─ ${OAUTH2_TOKEN_URL} set?  → oauth2
         ├─ ${BEARER_TOKEN}     set?  → jwt
         └─ neither                   → basic (default)
```

### The 4 supported methods

| Method | Required env vars | Typical use |
|--------|-------------------|-------------|
| **basic** | `ORG_ADMIN_USER`, `ORG_ADMIN_PASSWORD` | Local dev, internal envs |
| **jwt** | `BEARER_TOKEN` | CI pipelines, short-lived tokens |
| **oauth2** | `OAUTH2_TOKEN_URL`, `OAUTH2_CLIENT_ID`, `OAUTH2_CLIENT_SECRET` | Customer-managed identity |
| **sltoken** | `ORG_ADMIN_USER`, `ORG_ADMIN_PASSWORD` | SnapLogic-issued tokens |

---

## 5. Step 3 — Validate Environment Variables

This step **fails fast**. Before talking to any external service, it checks that every required variable is non-empty. If anything is missing, the suite stops with a clear error pointing you at the problem.

### Common required variables (always)

| Variable | Used for |
|----------|----------|
| `URL` | SnapLogic control plane URL (e.g. `https://elastic.snaplogic.com`) |
| `ORG_NAME` | SnapLogic organisation |
| `PROJECT_SPACE` | Project space to create / use |
| `PROJECT_NAME` | Project to create inside the space |
| `GROUNDPLEX_NAME` | Groundplex node where pipelines run |

### Auth-method-specific extras

| If AUTH_METHOD = | Also required |
|------------------|---------------|
| **basic** / **sltoken** | `ORG_ADMIN_USER` + `ORG_ADMIN_PASSWORD` |
| **jwt** | `BEARER_TOKEN` |
| **oauth2** | `OAUTH2_TOKEN_URL` + `OAUTH2_CLIENT_ID` + `OAUTH2_CLIENT_SECRET` |

> ⚠️ **If a variable is missing** the suite fails with a message like:
>
> *"Missing required environment variables for AUTH_METHOD=basic: ORG_NAME, PROJECT_SPACE. Please check your .env file..."*
>
> This is intentional. It's much better to fail at startup than mid-test with an obscure HTTP error.

## 6. Step 4 — Set Up Global Variables

This step turns local paths into `Set Global Variable`s, making them available to every test case in every sub-directory without re-importing.

| Variable | Resolves to | Used by |
|----------|-------------|---------|
| `${account_payload_path}` | `test/suite/test_data/accounts_payload` | Account-creation tests |
| `${env_file_path}` | `.env` at project root | SnapLogic API setup |
| `${pipeline_payload_path}` | `src/pipelines` | Pipeline import tests |
| `${generative_slp_pipelines_path}` | `src/generative_pipelines` | SLIM-generated pipelines |

## 7. Step 5 — Project Setup (the SnapLogic API call)

This is where the framework actually **talks to SnapLogic**. Based on the detected auth method, it dispatches to one of four branches and ultimately calls `Set Up Data` (a keyword from the `snaplogic_common_robot` library).

### What `Set Up Data` does internally

Behaviour depends on whether you passed `PROJECT_SPACE_SETUP=True` on the command line:

| Flag | Behaviour |
|------|-----------|
| `PROJECT_SPACE_SETUP=True` | 1. **Delete** the existing project space (if present)<br>2. **Create** a new project space named `${PROJECT_SPACE}`<br>3. **Create** a project inside it named `${PROJECT_NAME}`<br>4. **Set up** the Snaplex pointer to `${GROUNDPLEX_NAME}`<br>5. Pre-create the **shared accounts directory** (where the test framework will later POST account JSON) |
| `PROJECT_SPACE_SETUP=False` *(default)* | **Verify** the project space exists. If missing, **fail** with a clear error. Nothing is created or deleted. |

### Visual: PROJECT_SPACE_SETUP=True flow

```
1. Robot calls Set Up Data with the URL, credentials, and target
   project-space name from .env
        ↓
2. Library auths against the SnapLogic API using the detected method
        ↓
3. Issues DELETE for the existing project space
   (idempotent — succeeds even if absent)
        ↓
4. Issues POST to create a fresh project space
        ↓
5. Issues POST to create the project inside the space
        ↓
6. Sets up Groundplex binding so pipelines can execute
        ↓
7. Pre-creates shared/ folder for accounts
   (used by per-test "Create Account" cases)
        ↓
✓ Suite Setup complete. Test cases now run, each calling
  Create Account From Template etc. against this fresh project space.
```

> ⚠️ **When to use `PROJECT_SPACE_SETUP=True`:**
> - Very first run after pulling the repo
> - Snaplex configuration changed
> - Project space is corrupted or you want a clean slate
> - Always in CI/CD pipelines — gives a deterministic starting point
>
> **When to skip it (default `False`):**
> - You're iterating on tests and want to keep accounts/pipelines around
> - Quick smoke tests that should fail loudly if the project is missing

### Practical command-line invocations

```bash
# First time / CI — full setup
make robot-run-all-tests TAGS="sqlserver2" PROJECT_SPACE_SETUP=True

# Re-run on existing project space
make robot-run-tests-no-gp TAGS="sqlserver2"

# Use stage-specific overrides (loads .env.stage as Layer 3)
make robot-run-all-tests TAGS="sqlserver2" ENV=.env.stage
```

### Switching environments with `ENV=`

The `ENV=<file>` flag is the right way to point one test run at a different SnapLogic org, project space, or database without editing your `.env` file. The override file is loaded as **Layer 3** (highest priority) — anything defined in it wins over the same variable in `.env` or `env_files/`.

> ⚠️ **Important nuance:** `ENV=.env.stage` does **not** skip your `.env` file — both are loaded; the override just wins on conflicts. To effectively "ignore" `.env`, put every variable you care about into the override file. Anything not redefined there will fall back to `.env`.

#### Use case 1 — Run the same tests against multiple environments

Keep one credentials file per environment, switch on the command line:

```bash
# File: .env.stage
URL=https://stage.snaplogic.com
ORG_NAME=my-org-stage
PROJECT_SPACE=stage_pipeline_tests
ORG_ADMIN_USER=stage_user
ORG_ADMIN_PASSWORD=stage_pass

# File: .env.prod
URL=https://elastic.snaplogic.com
ORG_NAME=my-org-prod
PROJECT_SPACE=prod_pipeline_tests
ORG_ADMIN_USER=prod_user
ORG_ADMIN_PASSWORD=prod_pass
```

Then run either with:

```bash
# Run against stage
make robot-run-tests-no-gp TAGS="sqlserver2" ENV=.env.stage

# Run against prod
make robot-run-tests-no-gp TAGS="sqlserver2" ENV=.env.prod

# Run with default .env (no override)
make robot-run-tests-no-gp TAGS="sqlserver2"
```

#### Use case 2 — Override a single variable for one test run

Create a tiny override file with only what you want to change:

```bash
# File: .env.testdb
SQLSERVER_DATABASE=TEST_BACKUP
```

```bash
make robot-run-tests-no-gp TAGS="sqlserver2" ENV=.env.testdb
```

Everything else (URL, org, credentials, hosts) comes from your normal `.env`. Only `SQLSERVER_DATABASE` is swapped.

#### Use case 3 — CI/CD with injected secrets

In CI, the runner usually writes secrets into a temporary file and points the test at it:

```yaml
# GitHub Actions / GitLab CI / Travis snippet
- name: Run pipeline tests
  run: |
    cat > /tmp/.env.ci <<EOF
    URL=${SECRET_SNAPLOGIC_URL}
    ORG_ADMIN_USER=${SECRET_USER}
    ORG_ADMIN_PASSWORD=${SECRET_PASS}
    ORG_NAME=${SECRET_ORG}
    PROJECT_SPACE=ci_${BUILD_ID}
    EOF
    make robot-run-all-tests TAGS="sqlserver2" ENV=/tmp/.env.ci PROJECT_SPACE_SETUP=True
```

The repo's committed `.env` stays a placeholder; real credentials never land on disk outside the CI runner.

#### How to verify the override actually loaded

When the suite starts, `__init__.robot` prints the load order to the console:

```
Loading 4 environment files from /app/env_files/database_accounts:
Loading: .env.sqlserver
Loading: .env.oracle
...
Loading root .env file:
Loaded environment variables from: /app/.env

========================================
Loading ENV override file (HIGHEST PRECEDENCE):
/app/.env.stage
========================================
Loaded environment variables from: /app/.env.stage
```

If you don't see the "ENV override file" block, the flag wasn't picked up — check your `make` command spelling.

## 8. The two main Make targets

The Robot suite is invoked through one of two Make targets. Both ultimately call `robot-run-tests`, but they differ in **how much they manage around it**:

| Target | Manages Groundplex? | Use when |
|--------|--------------------|----------|
| `robot-run-all-tests` | Yes — starts and stops a local Docker Groundplex | Local dev; one command brings everything up |
| `robot-run-tests-no-gp` | Never — assumes the Groundplex is already running (or external) | Running against a cloud / external plex, or you started one yourself |

### Workflow: `robot-run-all-tests`

```
make robot-run-all-tests TAGS="sqlserver2"
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 0a — External-plex guard                             │
│    ✗ EXTERNAL_GROUNDPLEX=yes in .env?                       │
│      └─ FAIL with "use robot-run-tests-no-gp instead"       │
│    ✗ No local container AND no .slpropz?                    │
│      └─ ⚠ Warning, sleep 5s, continue (likely first run)    │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 0 — check-groundplex-swap                            │
│    Confirm Groundplex name didn't change underneath us.     │
│    Prevents accidentally swapping into someone else's plex. │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 1 — Project space + plex creation                    │
│                                                             │
│   PROJECT_SPACE_SETUP=True (default)                        │
│   ──────────────────────────────────                        │
│   Run: robot-run-tests TAGS="createplex"                    │
│    • Creates project space (idempotent)                     │
│    • Creates project inside it                              │
│    • Registers Snaplex definition                           │
│    • Downloads .slpropz                                     │
│                                                             │
│   On failure → check log for "active nodes" error:          │
│     ├─ YES → make stop-groundplex → wait 60s → retry        │
│     └─ NO  → exit 1                                         │
│                                                             │
│   PROJECT_SPACE_SETUP=False                                 │
│   ─────────────────────────                                 │
│   Run: robot-run-tests TAGS="verify_project_space_exists"   │
│    • Just checks it's there → fail if missing               │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 2 — Launch Groundplex                                │
│   make launch-groundplex SWAP_CHECK_DONE=1                  │
│    • docker-compose up the snaplogic-groundplex container   │
│    • Mounts .slpropz, joins snaplogicnet network            │
│    • Waits for plex to come online                          │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 2.1 — Travis permissions (CI only)                   │
│   if $TRAVIS == "true": chmod test_data/                    │
│   Otherwise: skip                                           │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 3 — Run user-defined tests                           │
│   robot-run-tests TAGS="sqlserver2" PROJECT_SPACE_SETUP=False
│   (Setup already done in Phase 1; don't re-do it.)          │
└─────────────────────────────────────────────────────────────┘
```

> **Use when:** local dev, you want one command to bring everything up.
> **Cannot use when:** `EXTERNAL_GROUNDPLEX=yes` in `.env`.

### Workflow: `robot-run-tests-no-gp`

```
make robot-run-tests-no-gp TAGS="sqlserver2"
            │
            ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 0 — Skipped intentionally                            │
│   No swap-check. Reason: "-no-gp" contract = framework      │
│   doesn't manage your Groundplex, especially externals.     │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 1 — Project space (no plex)                          │
│                                                             │
│   PROJECT_SPACE_SETUP=True (default)                        │
│   ──────────────────────────────────                        │
│   Run: robot-run-tests TAGS="verify_project_space_exists"   │
│         PROJECT_SPACE_SETUP=True                            │
│    • Idempotent project space + project create              │
│    • NO Snaplex registration (that's the point of -no-gp)   │
│                                                             │
│   On failure → "active nodes" error?                        │
│     ├─ YES, local container exists → stop, wait 60s, retry  │
│     │      (leaves GP STOPPED; user restarts manually)      │
│     ├─ YES, NO local container → external GP → fail with    │
│     │      "ACTION REQUIRED: stop it externally"            │
│     └─ NO  → exit 1                                         │
│                                                             │
│   PROJECT_SPACE_SETUP=False                                 │
│   ─────────────────────────                                 │
│   Run: robot-run-tests TAGS="verify_project_space_exists"   │
│         PROJECT_SPACE_SETUP=False                           │
│    • Verify-only → fail-fast if missing                     │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 2 — ⏩ SKIP all Groundplex/Plex operations           │
│   echo "Ensure Groundplex is already running."              │
│   (No launch, no stop, no .slpropz refresh.)                │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 2.1 — Travis permissions (CI only) — same as above   │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 2.5 — No-local-Groundplex warning                    │
│   docker ps | grep snaplogic-groundplex?                    │
│     ├─ FOUND → continue silently                            │
│     └─ NOT FOUND → ⚠ warn:                                  │
│         "If your tests need a plex, either:                 │
│           1. Run 'make launch-groundplex' in another tab    │
│           2. Ctrl+C and use 'make robot-run-all-tests'      │
│           3. Use an external Groundplex"                    │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  PHASE 3 — Run user-defined tests                           │
│   robot-run-tests TAGS="sqlserver2" PROJECT_SPACE_SETUP=False
│   (Same as robot-run-all-tests — only Phase 3 actually runs │
│    your tests; Phases 0–2.5 are setup/safety scaffolding.)  │
└─────────────────────────────────────────────────────────────┘
```

> **Use when:** external Groundplex already exists, OR you launched the local one yourself with `make launch-groundplex`.
> **Skips:** all Groundplex lifecycle management. Will warn (not fail) if no plex is running.

### Side-by-side phase comparison

| Phase | `robot-run-all-tests` | `robot-run-tests-no-gp` |
|-------|----------------------|-------------------------|
| 0a — External-plex guard | ✅ blocks if external | ❌ omitted |
| 0 — Swap check | ✅ runs | ❌ omitted by design |
| 1 — Project space | ✅ `createplex` | ✅ `verify_project_space_exists` w/ create |
| 2 — Launch Groundplex | ✅ `launch-groundplex` | ❌ skipped — your job |
| 2.1 — Travis perms | ✅ | ✅ |
| 2.5 — Plex-running warning | n/a (just launched it) | ✅ warns if missing |
| 3 — Run user tests | ✅ | ✅ |

### The two safety checks in detail

#### External-plex guard (Phase 0a)

Reads `EXTERNAL_GROUNDPLEX` from `.env`. If set to `yes` or `true`, the target **refuses to run** and points you at `robot-run-tests-no-gp` instead.

> ⚠️ **Why it exists — the footgun it prevents:**
>
> Imagine a customer's Groundplex called `prod-plex` running in their cloud, managed by their ops team. You set `GROUNDPLEX_NAME=prod-plex` in `.env` and run `make robot-run-all-tests`.
>
> Without the guard, Phase 2 would start a **local** Docker container also named `prod-plex` that registers itself with the customer's org. Two Groundplexes fighting over the same registration → their ops team gets paged.
>
> The guard makes you set `EXTERNAL_GROUNDPLEX=yes` as an explicit "don't touch" signal.

| Setting in `.env` | Behavior |
|-------------------|----------|
| `EXTERNAL_GROUNDPLEX=yes` | **Hard fail** — must use `-no-gp` target |
| No flag, no local artifacts | **Soft warn** — 5-second sleep, continue (likely first run) |
| No flag, local artifacts exist | Silent continue — looks like a normal repeat run |

#### Swap check (Phase 0)

Calls `make check-groundplex-swap`. Compares the `GROUNDPLEX_NAME` in your current `.env` against any local artifacts (the `snaplogic-groundplex` container and the `.slpropz` config file). If they don't match, it stops and asks for confirmation.

> ⚠️ **Why it exists — the footgun it prevents:**
>
> **Day 1:** `GROUNDPLEX_NAME=test_plex` in `.env`. Run tests. Container + `.slpropz` stay on disk for `test_plex`.
>
> **Day 2:** Co-worker hands you a different `.env` with `GROUNDPLEX_NAME=customer_prod`. You run `make robot-run-all-tests` without thinking.
>
> Without the swap check, Phase 2 might start your old `test_plex` container and register it under the new project space, OR fail in confusing ways because the `.slpropz` on disk doesn't match the plex you think you're using.

**Why `robot-run-tests-no-gp` skips this check:** if the framework isn't going to touch the Groundplex anyway, warning about a name swap would just be noise. The `-no-gp` contract is "framework doesn't manage your plex" — that includes not policing your config.

### Both targets ultimately call the same workhorse

The actual Robot command lives in `robot-run-tests`. Both wrappers eventually invoke it as their final step (Phase 3):

```bash
$(MAKE) robot-run-tests TAGS="$(TAGS)" PROJECT_SPACE_SETUP=False
```

which expands inside the `tools` Docker container to:

```bash
robot --variable PROJECT_SPACE_SETUP:False \
      --variable FORCE_RECREATE_PROJECT_SPACE:False \
      --variable TAGS:"sqlserver2" \
      --include sqlserver2 \
      --outputdir robot_output \
      suite/
```

Phase 3 in both targets is the same call. Everything in earlier phases is preparation: ensuring the project space exists, the Groundplex is reachable, and (in the all-tests case) the local plex is running.

## 9. TL;DR cheat sheet

> **The whole flow in 30 seconds:**
>
> 1. Robot auto-finds `__init__.robot` when you run any test
> 2. It loads `.env` in 3 layers (env_files/ → root .env → ENV= override)
> 3. It picks an auth method (auto or explicit)
> 4. It validates required env vars exist (fails fast if not)
> 5. It exposes paths as global variables
> 6. It calls SnapLogic to set up (or verify) the project space
> 7. Only then do your actual test cases run

### Files referenced in this walkthrough

| Path | Purpose |
|------|---------|
| `test/suite/__init__.robot` | The file this guide explains |
| `.env` | Root-level credentials and overrides (Layer 2) |
| `env_files/` | Per-service defaults (Layer 1) |
| `test/suite/test_data/accounts_payload/` | JSON templates for SnapLogic accounts |
| `src/pipelines/` | The .slp pipeline files |

### Common failure modes

| Symptom | Likely cause |
|---------|--------------|
| `Missing required environment variables for AUTH_METHOD=basic: ...` | You forgot to fill in `.env` — go back to step 3 |
| `Project space 'X' is not created` | You ran without `PROJECT_SPACE_SETUP=True` on first run |
| `Connection refused` on first API call | `URL` in .env is wrong, or VPN/firewall |
| Variables silently empty | JSON parse error in `.env` — check for unmatched quotes / brackets |
