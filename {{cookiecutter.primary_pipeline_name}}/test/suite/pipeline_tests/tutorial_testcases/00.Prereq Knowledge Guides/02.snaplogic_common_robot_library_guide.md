# SnapLogic Common Robot Library Guide

## What is this library? вҖ” a layman's introduction

Before diving into the technical details, let's understand **what this library is and why it exists** вҖ” using a comparison most people already understand.

### The Python / Java library analogy

Think about how you use any popular **Python library** вҖ” say, `requests` for HTTP calls:

```python
# Without the library вҖ” you'd have to do all this manually:
import socket
# open a TCP connection, format an HTTP request by hand,
# parse the response bytes, handle redirects, decode JSON...
# (hundreds of lines of low-level code)

# WITH the requests library вҖ” one line:
response = requests.get("https://api.example.com/users")
```

Someone smart already **wrapped all those low-level details** into a clean function called `get()`. You don't care *how* it works under the hood вҖ” you just call it and get your result. Same story with Java's `HttpClient`, `Jackson`, or any well-built library.

> **A library = pre-built functions that hide the messy details, so you only deal with the easy outer layer.**

### Now apply the same idea to SnapLogic

In SnapLogic, **every action is controlled by an API call**. For example, to create an account:

- You need to know the right API endpoint
- You need to know whether it's a `POST` or `GET`
- You need to build the right JSON payload
- Sometimes you need 2вҖ“3 API calls in sequence (create вҶ’ validate вҶ’ assign)
- You need to handle auth tokens, error responses, retries...

If you were writing tests *without* a library, every single test would have to do all of that by hand. That's painful, error-prone, and not what a tester should be spending time on.

### Enter `snaplogic_common_robot`

This library does for **SnapLogic testing** what `requests` does for HTTP: it **wraps all the underlying API calls into simple, readable keywords**.

### Side-by-side comparison

| Layer                      | Python ecosystem           | Java ecosystem          | SnapLogic + Robot Framework               |
| -------------------------- | -------------------------- | ----------------------- | ----------------------------------------- |
| **Low-level primitive**    | `socket`, raw HTTP         | `HttpURLConnection`     | Raw SnapLogic REST API calls              |
| **Curated library on top** | `requests`, `pandas`       | `HttpClient`, `Jackson` | **`snaplogic_common_robot`**              |
| **What the user writes**   | `requests.get(url)`        | `client.send(request)`  | `Create Account From Template`            |
| **What's hidden**          | Sockets, encoding, parsing | Streams, marshalling    | API endpoints, payloads, auth, sequencing |

### What's actually inside this library?

A **vast collection of 400+ pre-built keywords** covering nearly every SnapLogic operation you'd ever need in a test:

- `Create Account From Template` вҖ” wraps account creation API calls
- `Import Pipeline` вҖ” wraps pipeline upload + validation
- `Create Triggered Task` вҖ” wraps task setup
- `Execute Triggered Task` вҖ” wraps task execution + result polling
- `Upload File To SLDB` вҖ” wraps file upload API
- `Create Project Space` / `Delete Project Space` вҖ” wraps project-space lifecycle
- ...and 390+ more

Each keyword internally handles the right API endpoint, the right payload shape, the right auth, the right error handling вҖ” so you don't have to.

### How do I see all the available keywords?

Two easy ways:

```bash
# Option 1: Generate the keyword docs locally
# вҡ пёҸ Prerequisite: the tools container must be running first
make start-tools-service-only   # one-time, if not already running
make show-libdocs                # extracts libdocs HTML to your workspace

# Option 2: Browse the installed package
# (just like you'd browse a Python package in site-packages)
cd snaplogic-common-robot/src/snaplogic_common_robot/
```

The first option creates an HTML file you can open in a browser and search through all 400+ keywords with their descriptions, arguments, and examples.

> **Heads up:** if you run `make show-libdocs` without starting the tools container first, you'll see:
> ```
> вқҢ Error: Container 'snaplogic-test-example-tools-container' is not running!
> рҹ’Ў Start it first with: make start-tools-service-only
> ```
> This is because the library is installed *inside* the Docker container, not on your host machine. The `show-libdocs` target copies the libdocs out of the container вҖ” so the container has to be alive when you run it.

### How does my test use it?

You just **import the library** at the top of your Robot Framework test file, and all 400+ keywords become available вҖ” exactly like `import requests` in Python:

```robot
*** Settings ***
Library    snaplogic_common_robot.snaplogic_keywords

*** Test Cases ***
Create Oracle Account
    Create Account From Template
    ...    account_payload_file=acc_oracle.json
    ...    account_name=oracle_acct
```

If you look at any test case in this framework, you'll see they're **almost entirely calls to keywords from this library**. The library does the heavy lifting; your test case just describes *what* you want to verify, not *how* to talk to SnapLogic.

### The big picture рҹ’Ў

> **`snaplogic_common_robot` exists so you write tests in plain English, not in REST API calls.**
>
> The library team's job is to keep improving these keywords. Your job as a test author is to pick the right keywords, pass the right arguments, and assert the right results. Everything in between is handled for you.

The rest of this guide explains the technical details: how the library is packaged, installed, structured, and how to extend it.

## Overview

**snaplogic_common_robot** is a custom Robot Framework library that provides reusable keywords and utilities specifically designed for SnapLogic automation testing. This library encapsulates common SnapLogic operations, API interactions, and testing patterns into convenient Robot Framework keywords.

### Key Features

- **SnapLogic API Integration** - Keywords for SnapLogic REST API operations
- **Pipeline Management** - Keywords for pipeline import, export, and execution
- **Account Management** - Keywords for creating and managing SnapLogic accounts
- **Project Operations** - Keywords for project space and project management
- **Groundplex Operations** - Keywords for Groundplex creation and management
- **Embedded Documentation** - Complete keyword documentation included in the package
- **Reusable Components** - Common testing patterns abstracted into keywords

### Library Benefits

- **Consistency** - Standardized approach to SnapLogic testing across projects
- **Productivity** - Pre-built keywords reduce test development time
- **Maintainability** - Centralized library simplifies updates and bug fixes
- **Documentation** - Built-in keyword documentation and examples
- **Version Control** - Semantic versioning for stable releases
- **Rich Dependencies** - Includes comprehensive set of testing libraries automatically

## Installation Process

### Installation via requirements.txt

The library is installed automatically as part of the Docker container build process. When you install snaplogic-common-robot, it automatically installs all its dependencies:

```txt
# requirements.txt
snaplogic-common-robot
# This single line automatically installs 20+ packages
```

### Container Build Integration

```dockerfile
# robot.Dockerfile
FROM python:3.11-slim

# Set the working directory
WORKDIR /app
# Copy the requirements file and install dependencies
COPY requirements.txt . 
RUN pip install --no-cache-dir --index-url https://pypi.org/simple -r requirements.txt

# The snaplogic-common-robot library is now available
```

### Manual Installation

For local development or exploration:

```bash
# Install from PyPI (includes ALL dependencies automatically)

# Install specific version
pip install snaplogic-common-robot==1.2.3
```
